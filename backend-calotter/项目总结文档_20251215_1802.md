# 项目总结文档

**更新时间**: 2025-12-15 18:02

## 📋 项目概述

本项目是一个基于 Spring Boot 的智能烹饪助手后端系统，采用模块化设计，主要包含用户管理、库存管理、烹饪推荐和健康管理四大核心功能模块。

## 📦 已实现的实体类（Entity）

### 1. 基础实体类

#### BaseEntity
- **位置**: `calotter-common/src/main/java/com/calotter/common/core/domain/BaseEntity.java`
- **功能**: 所有实体类的基类
- **字段**:
  - `createTime` - 创建时间（自动填充）
  - `updateTime` - 更新时间（自动填充）
  - `createBy` - 创建人ID（Long类型）
  - `updateBy` - 更新人ID（Long类型）
- **特性**: 使用 JPA Auditing 自动管理审计字段

### 2. 标准数据字典（calotter-common 模块）

#### RefAllergen
- **表名**: `ref_standard_allergens`
- **功能**: 标准过敏原字典
- **字段**:
  - `id` - 主键
  - `name` - 过敏原名称（唯一）
  - `description` - 描述/严重程度备注

#### StandardIngredient
- **表名**: `ref_standard_ingredients`
- **功能**: 标准食材库
- **字段**:
  - `id` - 主键（手动分配）
  - `name` - 食材名称
  - `category` - 分类（MEAT, VEG等）
  - `calories`, `protein`, `fat`, `carb`, `fiber` - 营养素（每100g）
  - `averageGramPerUnit` - 单位换算（每单位对应的克数）
  - `shelfLifePantry/Fridge/Freezer` - 保质期（天）
  - `defaultLocation` - 默认存储位置（枚举）
- **关系**: 多对多关联 `RefAllergen`（包含的过敏原）

#### StandardSpice
- **表名**: `ref_standard_spices`
- **功能**: 标准调料库
- **字段**:
  - `id` - 主键
  - `name` - 调料名称
- **关系**: 多对多关联 `RefAllergen`（包含的过敏原）

#### StandardUtensil
- **表名**: `ref_standard_utensils`
- **功能**: 标准厨具库
- **字段**:
  - `id` - 主键（手动分配，便于图标映射）

  - `iconUrl` - 图标资源路径

### 3. 用户模块实体（calotter-user 模块）

## 🔧 已实现的业务逻辑

### 1. 用户认证服务（UserService）

#### 功能概述
提供用户注册和登录功能，包含密码加密和 JWT Token 生成。

#### 核心方法

**register(RegisterRequest request)**
- **功能**: 用户注册
- **流程**:
  1. 检查用户名唯一性
  2. 检查邮箱唯一性
  3. 使用 `PasswordEncoder` 加密密码
  4. 创建用户实体（默认角色ROLE_USER，状态可用，未完成引导）
  5. 保存用户
  6. 生成 JWT Token
  7. 返回认证响应（包含Token和用户信息）
- **异常处理**: 用户名或邮箱已存在时抛出 `IllegalArgumentException`

**login(LoginRequest request)**
- **功能**: 用户登录
- **流程**:
  1. 支持用户名或邮箱登录（优先用户名，失败则尝试邮箱）
  2. 验证密码（使用 `PasswordEncoder.matches`）
  3. 检查用户状态（未激活或封禁时拒绝登录）
  4. 生成 JWT Token
  5. 返回认证响应
- **异常处理**: 用户名/密码错误或账户状态异常时抛出 `IllegalArgumentException`

### 2. JWT Token 服务（JwtService）

#### 功能概述
提供 JWT Token 的生成、解析和验证功能。

#### 核心方法

**generateToken(Long userId, String username)**
- 生成包含 userId 和 username 的 JWT Token
- 配置了 issuer、audience 和过期时间

**extractUsername(String token)**
- 从 Token 中提取用户名

**extractUserId(String token)**
- 从 Token 中提取用户ID

**validateToken(String token)**
- 验证 Token 是否有效且未过期

### 3. 烹饪上下文构建服务（CookingContextBuilderService）

#### 功能概述
构建用于 AI 烹饪推荐的完整上下文，整合家庭成员信息、健康目标、库存状态等多维度数据。

#### 核心方法

**buildContext(CookingGenerationRequest req)**
- **功能**: 构建 AI 请求上下文的主入口
- **流程**:
  1. 根据成员ID列表获取家庭成员数据
  2. 调用 `processDinersAndGoals` 处理用餐者画像与营养目标
  3. 调用 `processInventory` 处理厨房库存
  4. 构建任务设置（菜品数量、最大时间、难度、目标菜系）
  5. 组装并返回 `AiCookingContext` 对象

**processDinersAndGoals(...)**
- **功能**: 处理所有用餐者的画像与营养目标
- **核心逻辑**:
  1. **处理家庭成员**:
     - 收集硬性限制（过敏原、饮食风格）加入全局避让列表
     - 统计软性偏好（忌口、口味、菜系）频率
     - 提取个人忌口用于个人画像
     - 根据健康目标计算营养需求（本餐按35%比例分配，无目标则使用默认值）
     - 构建成员花名册（DinerSlot）
  2. **处理客人**:
     - 收集过敏原和偏好
     - 使用默认营养值（700卡路里）
     - 加入花名册
  3. **冲突分析**:
     - 分析忌口冲突（所有人都不吃→全局屏蔽，部分人不吃→记录冲突）
     - 分析菜系偏好（半数以上喜欢→共同喜好）
  4. **营养累加**:
     - 汇总所有用餐者的营养目标（卡路里范围、蛋白质、脂肪、碳水、纤维）
  5. **返回**: `DinerProfile` 和 `CompositeNutritionalGoal`

**processInventory(Long householdId)**
- **功能**: 处理厨房库存快照
- **核心逻辑**:
  1. **处理食材**:
     - 查询数量大于0的食材
     - 计算剩余保质期（距离过期日期的天数）
     - 分类处理：
       - 3天内过期 → 标记为 URGENT，加入优先级列表
       - 其他 → 标记为 NORMAL，加入普通列表
       - 无过期日期 → 标记为 NORMAL
  2. **处理调料**:
     - 查询可用（isAvailable=true）的调料
     - 提取调料名称列表
  3. **处理厨具**:
     - 查询可用（isAvailable=true）的厨具
     - 提取厨具名称列表
  4. **返回**: `KitchenSnapshot`（包含优先级食材、普通食材、可用调料、可用厨具）

### 4. Dish服务（DishService）✅ 新增

#### 功能概述
负责从AI响应创建Dish快照，实现Dish实体与AiRecipeResponse之间的转换。

#### 核心方法

**createDishFromAiResponse(AiRecipeResponse aiResponse, Household household)**
- **功能**: 从AI响应创建Dish快照
- **流程**:
  1. 验证AI响应数据完整性
  2. 提取主菜信息（取第一个菜品）
  3. 映射基础信息（name, description, cookingTimeMinutes）
  4. 转换难度等级（字符串转枚举，无效值默认MEDIUM）
  5. 提取营养信息（从totalNutrition）
  6. 计算总重量（累加所有单位为"g"的食材重量，默认1000g）
  7. 映射烹饪步骤
  8. 映射原料快照
  9. 保存并返回Dish实体
- **异常处理**: AI响应数据不完整时抛出 `IllegalArgumentException`

### 5. 烹饪会话服务（CookingSessionService）✅ 新增

#### 功能概述
负责完成烹饪会话，处理剩菜，发布事件触发健康日志创建。

#### 核心方法

**completeSession(CookingCompletionRequest req)**
- **功能**: 完成烹饪会话，记录家庭成员摄入量，并自动处理剩菜
- **流程**:
  1. 获取Session和Household
  2. 创建或获取Dish快照（如果session.finalDish不存在，则从aiResponse创建）
  3. 计算总消费比例
  4. 处理剩菜（如果剩余比例>5%且用户选择保存，则创建LeftoverDish）
  5. 更新Session状态为COOKED
  6. 发布`CookingSessionCompletedEvent`事件（触发health模块创建NutritionLog）
  7. 计算并返回响应（包含日志数量、剩菜是否创建、总卡路里等）
- **返回**: `CookingCompletionResponse`
- **异常处理**: Session或Household不存在时抛出 `IllegalArgumentException`

### 6. 剩菜业务服务（LeftoverDishService）

#### 功能概述
处理剩菜相关业务逻辑，组合 LeftoverDish 和 Dish 的数据，提供查询、营养计算等功能。

#### 核心方法

**getLeftoverDishDetail(Long leftoverId)**
- **功能**: 根据ID查询剩菜详情
- **流程**:
  1. 查询 LeftoverDish 实体
  2. 通过 `originalDishId` 查询关联的 Dish 实体
  3. 计算当前剩菜的营养信息
  4. 组合数据构建并返回 `LeftoverDishDetailDTO`
- **返回**: `LeftoverDishDetailDTO`（包含完整信息和计算的营养数据）

**getLeftoverDishesByHousehold(Long householdId)**
- **功能**: 根据家庭ID批量查询剩菜列表（批量优化版本）
- **流程**:
  1. 批量查询 LeftoverDish（按家庭ID）
  2. 提取所有 `originalDishId`
  3. 批量查询 Dish（避免 N+1 问题，使用 Map 缓存）
  4. 组合数据到 DTO 列表
- **性能优化**: 使用批量查询和 Map 缓存，避免循环查询导致的 N+1 问题
- **返回**: `List<LeftoverDishSummaryDTO>`

**calculateNutritionForConsumption(Long leftoverId, Integer consumedGram)**
- **功能**: 计算指定重量的营养信息（用于食用剩菜）
- **流程**:
  1. 验证参数（食用重量必须大于0）
  2. 查询 LeftoverDish 实体
  3. 验证是否可以食用（食用重量不能超过剩余重量）
  4. 查询关联的 Dish 实体
  5. 根据比例计算营养信息
- **计算公式**: 营养 = (consumedGram / dish.totalWeightGram) * dish.totalNutrient
- **返回**: `NutritionInfo`（包含 calories, protein, fat, carb, fiber）

**canConsume(Long leftoverId, Integer consumedGram)**
- **功能**: 验证是否可以食用指定重量的剩菜
- **返回**: true 如果可以食用，false 否则

### 7. 营养日志服务（NutritionLogService）✅ 新增

#### 功能概述
负责创建营养日志记录，支持从烹饪完成事件、剩菜和手动输入三种方式创建。

#### 核心方法

**createFromEvent(CookingSessionCompletedEvent event)**
- **功能**: 从烹饪完成事件创建营养日志（由事件监听器调用）
- **流程**:
  1. 转换MealType字符串为枚举
  2. 为每个用餐者创建NutritionLog
  3. 基于事件中的Dish营养快照计算并存储快照值
  4. 保存所有日志
  5. 发布`NutritionLogCreatedEvent`事件（触发聚合表更新）
- **返回**: `List<NutritionLog>`
- **异常处理**: 家庭成员不存在时抛出 `IllegalArgumentException`

**createFromLeftover(Long leftoverId, Long memberId, Integer consumedGram, LocalDateTime eatenAt)**
- **功能**: 从剩菜创建营养日志
- **流程**:
  1. 查询LeftoverDish
  2. 先验证食用重量（null检查和有效性验证，早验证早失败，避免不必要的服务调用）
  3. 验证食用重量是否超过剩余重量
  4. 查询家庭成员
  5. 使用LeftoverDishService计算营养
  6. 获取剩菜详情（包含Dish信息）
  7. 创建NutritionLog（使用弱引用dishId）
  8. 保存日志
  9. 更新LeftoverDish的剩余重量
  10. 发布事件触发聚合表更新
- **返回**: `NutritionLog`
- **异常处理**: 
  - 剩菜不存在时抛出 `IllegalArgumentException`
  - 食用重量为null或小于等于0时抛出 `IllegalArgumentException("食用重量必须大于0")`
  - 食用重量超过剩余重量时抛出 `IllegalArgumentException`（包含详细的重量信息）
  - 成员不存在时抛出 `IllegalArgumentException`

**createManual(ManualNutritionLogRequest request)**
- **功能**: 手动创建营养日志
- **流程**:
  1. 查询家庭成员
  2. 创建NutritionLog（dishId为null）
  3. 设置营养数据（来自请求）
  4. 保存日志
  5. 发布事件触发聚合表更新
- **返回**: `NutritionLog`
- **异常处理**: 家庭成员不存在时抛出 `IllegalArgumentException`

### 8. 营养聚合服务（NutritionAggregateService）✅ 新增

#### 功能概述
负责更新和查询日营养聚合表，生成周健康报告。

#### 核心方法

**updateAggregate(NutritionLog log)**
- **功能**: 更新聚合表（通过事件监听器调用）
- **流程**:
  1. 获取或创建聚合记录（如果不存在则创建新记录）
  2. 如果创建新记录，快照当天的健康目标（如果存在）
  3. 累加营养值（calories, protein, fat, carb, fiber）
  4. 保存聚合记录
- **特性**: 使用乐观锁（@Version）防止并发写入导致统计错误

**getWeeklyReport(Long memberId, LocalDate weekStart)**
- **功能**: 获取周健康报告
- **流程**:
  1. 获取家庭成员
  2. 获取健康目标
  3. 获取实际摄入（查询聚合表，日期范围查询）
  4. 计算本周总摄入（累加7天的聚合数据）
  5. 计算本周总目标（日目标 * 7）
  6. 构建每日详情（包含每日目标和实际值）
  7. 组装并返回WeeklyReportVO
- **返回**: `WeeklyReportVO`（包含周目标、周实际、每日详情）
- **异常处理**: 家庭成员不存在时抛出 `IllegalArgumentException`

**getOrCreateDailyAggregate(FamilyMember member, LocalDate date)**
- **功能**: 获取或创建日聚合记录
- **返回**: `DailyNutrientAggregate`

## 📝 核心业务场景

### 场景1: 用户注册和登录
1. 用户通过 `/api/user/register` 注册账户
2. 系统验证用户名和邮箱唯一性
3. 密码加密后保存
4. 生成并返回 JWT Token
5. 用户使用 Token 进行后续请求认证

### 场景2: 智能烹饪推荐上下文构建
1. 用户发起烹饪请求（指定用餐成员、客人、偏好等）
2. 系统调用 `CookingContextBuilderService.buildContext` 构建上下文
3. 系统收集所有用餐者的：
   - 过敏原和饮食限制（硬性限制）
   - 口味和菜系偏好（软性偏好）
   - 健康目标和营养需求
4. 系统分析偏好冲突（全局避让 vs 部分冲突）
5. 系统查询厨房库存，分类处理临期食材
6. 系统汇总营养目标（累加所有用餐者需求）
7. 返回完整的 AI 烹饪上下文，用于后续 AI 推荐

### 场景3: 烹饪完成和营养记录 ✅ 新增
1. 用户完成烹饪后，调用 `/api/cooking/complete` 端点
2. 系统创建或获取Dish快照（如果不存在）
3. 系统根据用餐者比例计算个人摄入量
4. 系统处理剩菜（如果剩余比例>5%且用户选择保存）
5. 系统发布`CookingSessionCompletedEvent`事件
6. 事件监听器创建NutritionLog记录（异步）
7. 系统更新Session状态为COOKED
8. 营养日志创建后，触发聚合表更新（异步）

### 场景4: 剩菜管理和查询
1. 烹饪完成后，系统创建 Dish 快照（包含完整营养信息）
2. 创建 LeftoverDish 记录，通过 `originalDishId` 关联 Dish
3. 用户查询剩菜列表时，系统批量查询 Dish 并组合数据
4. 用户食用剩菜时，调用 `/api/nutrition/log/leftover` 端点
5. 系统先验证食用重量（null检查和有效性验证）
6. 系统根据食用重量计算营养摄入
7. 系统创建NutritionLog记录
8. 系统更新LeftoverDish的剩余重量
9. 触发聚合表更新（异步）

### 场景5: 周健康报告查询 ✅ 新增
1. 用户调用 `/api/nutrition/weekly` 端点
2. 系统查询DailyNutrientAggregate表（日期范围查询，仅7条记录）
3. 系统获取健康目标（HealthGoal）
4. 系统计算周目标（日目标 * 7）
5. 系统累加周实际摄入（汇总7天的聚合数据）
6. 系统构建每日详情
7. 返回周报告（包含目标对比和每日详情）

### 场景6: 手动营养记录 ✅ 新增
1. 用户调用 `/api/nutrition/log/manual` 端点
2. 系统创建NutritionLog记录（dishId为null）
3. 系统保存营养数据（来自用户输入）
4. 触发聚合表更新（异步）
