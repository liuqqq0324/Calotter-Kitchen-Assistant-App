## 🔄 运行逻辑

### 用户信息获取流程

1. **前端请求**：`GET /api/ums/user?id={userId}`（或使用 JWT token）
2. **后端处理**：
   - 从 JWT token 或请求参数中提取用户ID
   - 从数据库获取用户信息
   - 从 `User.settings.profile` 中提取 profile 信息
   - 返回 `UserResponse`（包含 userId, userName, email, role, profile）
3. **前端解析**：从 `Result<UserResponse>` 的 `data` 字段中提取用户数据
4. **前端显示**：在 profile 页面显示用户信息

### 用户信息更新流程

1. **前端请求**：`PUT /api/ums/user?id={userId}`（包含 profile 数据）
2. **后端处理**：
   - 从 JWT token 或请求参数中提取用户ID
   - 更新 `User.settings.profile` 字段
   - 保存到数据库
   - 返回更新后的 `UserResponse`
3. **前端解析**：从 `Result<UserResponse>` 的 `data` 字段中提取更新后的数据
4. **前端刷新**：更新 profile 页面显示

### 用户偏好/禁忌/过敏获取流程

1. **前端请求**：`GET /api/ums/user/preferences?id={userId}`（或使用 JWT token）
2. **后端处理**：
   - 从 JWT token 或请求参数中提取用户ID
   - 从 `User.settings.preferences` 或 `User.settings.taboos` 中提取数据
   - 对于过敏，从 `FamilyMember.allergies` 中提取（ManyToMany 关系）
   - 返回相应的 Response DTO
3. **前端解析**：从 `Result<T>` 的 `data` 字段中提取数据
4. **前端显示**：在相应页面显示数据

### 用户偏好/禁忌/过敏更新流程

1. **前端请求**：`PUT /api/ums/user/preferences?id={userId}`（包含更新数据）
2. **后端处理**：
   - 从 JWT token 或请求参数中提取用户ID
   - 更新 `User.settings` 中的相应字段
   - 对于过敏，更新 `FamilyMember.allergies`（如果不存在则创建 FamilyMember）
   - 保存到数据库
   - 返回更新后的 Response DTO
3. **前端解析**：从 `Result<T>` 的 `data` 字段中提取更新后的数据
4. **前端刷新**：更新页面显示

## 🚨 重要问题和修复

### 问题 1: Profile 页面无法显示用户信息

**问题描述**：
- 前端用 testuser 或 chase 登录，profile 页面都显示 "Chase_70_70"
- Profile 编辑页面无法保存

**问题原因**：
1. 后端缺少 `GET /api/user` 和 `PUT /api/user` 端点
2. 前端 API 调用失败后回退到静态数据 `kCurrentUser`
3. 前端解析 `Result<T>` 格式不正确

**解决方案**：
1. ✅ 在 `UserService` 中添加了 `getUserInfo()` 和 `updateUserInfo()` 方法
2. ✅ 在 `UserController` 中添加了 `GET /api/user` 和 `PUT /api/user` 端点
3. ✅ 修复了前端 `user_service.dart` 中所有 API 方法的数据解析逻辑
4. ✅ 统一从 `Result<T>` 的 `data` 字段中提取实际数据

**修复结果**：
- ✅ Profile 页面现在可以正确显示实际登录用户的信息
- ✅ Profile 编辑页面可以正确保存用户信息

### 问题 2: 异常处理返回错误状态码

**问题描述**：
- `IllegalArgumentException` 返回 500 而不是 400
- `HttpMessageNotReadableException`（JSON 解析错误）返回 500 而不是 400

**解决方案**：
1. ✅ 修复了 `UserController` 中所有 `IllegalArgumentException` 捕获，显式返回 400
2. ✅ 在 `GlobalExceptionHandler` 中添加了 `HttpMessageNotReadableException` 处理

**修复结果**：
- ✅ 所有参数验证错误现在返回 400 状态码
- ✅ JSON 解析错误现在返回 400 状态码和友好消息

## 📊 技术要点总结

### 1. Result<T> 统一响应格式

- **格式**：`{code: 200, message: "操作成功", data: {...}}`
- **优势**：统一的响应格式，便于前端处理
- **注意**：前端需要从 `data` 字段中提取实际数据

### 2. JSONB 数据存储

- **User.settings**：使用 PostgreSQL JSONB 类型存储灵活的用户设置
- **优势**：无需修改表结构即可添加新字段
- **注意**：需要处理类型转换和空值检查

### 3. ManyToMany 关系处理

- **FamilyMember.allergies**：使用 ManyToMany 关系关联标准过敏原表
- **优势**：数据规范化，避免重复
- **注意**：需要确保标准过敏原数据存在

### 4. 路径兼容性

- **双路径支持**：`@RequestMapping({"/api/user", "/api/ums/user"})`
- **优势**：兼容前端现有路径，无需修改前端代码
- **建议**：未来统一使用 `/api/user/*` 路径

### 5. JWT Token 认证

- **用户ID提取**：支持从 JWT token 或请求参数中提取
- **优势**：灵活支持不同场景（前端调用、测试、管理员操作）
- **注意**：请求参数优先于 JWT token（用于测试或管理员操作）
