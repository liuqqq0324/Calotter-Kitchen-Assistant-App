## 📊 Preference 数据流和使用逻辑

### 核心概念

**用户偏好（User Preferences）** 是用户的**默认偏好设置**，存储在 `User.preferences` Map 中，包含两个大类：
- `TASTE` - 口味偏好（light, rich, spicy, sweet, sour, salty, umami）
- `CUISINE` - 菜系偏好（chinese, japanese, korean, etc.）

**烹饪过滤器（Recipe Generation Filter）** 是**临时的一次性设置**，用于单次菜单生成，可以覆盖用户的默认偏好。

### 数据流图

```
┌─────────────────────────────────────────────────────────────┐
│  User 模块：用户偏好管理（默认偏好）                          │
├─────────────────────────────────────────────────────────────┤
│  PreferencesListPage                                        │
│    ↓ 编辑保存                                                │
│  PUT /api/user/preferences-map                              │
│    ↓ 验证并保存                                              │
│  User.preferences Map {                                     │
│    "TASTE": ["spicy", "sweet"],                             │
│    "CUISINE": ["chinese", "japanese"]                       │
│  }                                                           │
└─────────────────────────────────────────────────────────────┘
                          ↓ 读取默认值
┌─────────────────────────────────────────────────────────────┐
│  Cooking 模块：菜单生成（临时过滤器）                        │
├─────────────────────────────────────────────────────────────┤
│  RecipeFilterPage                                           │
│    ↓ 加载默认值                                              │
│  GET /api/ai/default-filter                                 │
│    ↓ AiMenuService.getDefaultFilter()                       │
│    ↓ 从 User.preferences 读取                               │
│  RecipeGenerationFilter {                                   │
│    dietPreferences: {                                      │
│      tastePreferences: ["spicy", "sweet"],                 │
│      cuisinePreferences: ["chinese", "japanese"]          │
│    }                                                        │
│  }                                                           │
│    ↓ 用户可以在页面上临时修改                                │
│  RecipeGenerationFilter {                                   │
│    dietPreferences: {                                      │
│      tastePreferences: ["light", "sweet"],  ← 临时修改     │
│      cuisinePreferences: ["italian"]        ← 临时修改     │
│    }                                                        │
│  }                                                           │
│    ↓ 生成菜单                                                │
│  POST /api/ai/generate-menus                               │
│    ↓ AiMenuService.generateMenus()                          │
│    ↓ 直接转为 JSON 发送给 AI                                │
│  AI Prompt JSON (包含修改后的 filter)                       │
└─────────────────────────────────────────────────────────────┘
```

### 关键设计原则

1. **用户偏好 = 默认值**
   - `User.preferences` 存储用户的长期偏好设置
   - 作为 `RecipeGenerationFilter` 的初始值来源
   - 不会因为单次生成而改变

2. **Filter = 临时覆盖**
   - `RecipeGenerationFilter` 是单次生成时的临时设置
   - 用户可以临时修改，不影响 `User.preferences`
   - 直接作为 JSON 发送给 AI，不保存回数据库

3. **数据转换**
   - `PreferenceConverter.toDietPreferences()` - 将 `User.preferences` 转换为 `RecipeGenerationFilter.DietPreferences`
   - 转换是单向的（User → Filter），不会回写

### 使用场景

**场景 1：用户首次使用**
- 用户偏好为空
- Filter 页面显示空值
- 用户可以在 Filter 页面选择偏好并生成菜单
- 如果用户想保存为默认偏好，需要去 PreferencesListPage 保存

**场景 2：用户已有偏好**
- Filter 页面自动加载用户的默认偏好
- 用户可以临时修改（例如：今天想吃意大利菜）
- 生成菜单使用修改后的值
- 用户的默认偏好保持不变

**场景 3：用户更新偏好**
- 用户在 PreferencesListPage 编辑并保存
- 下次打开 Filter 页面时，会加载新的默认值

## 🔄 解决的问题

### 问题 1：没有统一的标准库 ✅ 已解决

**之前**：
- User 模块中只有注释说明 Key 的约定
- Cooking 模块中有私有常量定义
- 没有统一管理的地方

**现在**：
- 所有常量集中在 `PreferenceStandardLibrary` 中
- 所有模块都引用同一个标准库
- 预设值列表与前端保持一致

### 问题 2：字符串常量分散，容易出错 ✅ 已解决

**之前**：
- `CookingContextBuilderService` 中有私有常量
- `AiMenuService` 中直接使用硬编码字符串
- 容易拼写错误，难以维护

**现在**：
- 所有地方都使用 `PreferenceStandardLibrary.PREF_KEY_*` 常量
- 类型安全，编译时检查
- 修改时只需更新一处

### 问题 3：两个模块使用不同结构，但概念应一致 ✅ 已解决

**之前**：
- User 模块：`Map<String, List<String>> preferences`（Key: "TASTE", "CUISINE"）
- Cooking 模块：`RecipeGenerationFilter.DietPreferences`（字段：`tastePreferences`, `cuisinePreferences`）
- 没有明确的映射关系和转换方法

**现在**：
- 统一了 Key 常量，语义清晰
- 提供了 `PreferenceConverter` 工具类进行转换
- 两个结构之间的映射关系明确

### 问题 4：前端可以输入任意值，后端无验证 ✅ 已解决

**之前**：
- 前端 PreferencesListPage 允许自由输入
- 后端直接保存，无验证
- 数据不一致

**现在**：
- 前端使用 FilterChip 限制为标准库选项
- 后端添加验证，拒绝非标准库的值
- 前后端数据一致

### 问题 5：偏好分类不清晰 ✅ 已解决

**之前**：
- 混淆了 DISLIKE（不喜欢的食材）和 COOKER（厨具）的概念
- DISLIKE 不应该作为用户偏好

**现在**：
- 用户偏好只有两个大类：TASTE 和 CUISINE
- 厨具（COOKER）从家庭厨具表获取，不是用户偏好
- 不喜欢的食材（DISLIKE/avoidIngredients）应从 User.taboos 获取

## 🎯 技术要点

### 1. 模块依赖设计

- **calotter-common** 模块不依赖其他业务模块
- 标准库只包含常量和预设值，不包含业务逻辑
- 转换工具放在 `calotter-cooking` 模块中（因为它依赖 `RecipeGenerationFilter`）

### 2. 前后端一致性

- 预设值列表与前端 `recipe_filter_page.dart` 完全一致
- 确保前后端使用相同的选项值
- 前端使用 FilterChip 限制输入，后端验证确保数据一致性

### 3. 类型安全

- 使用常量而非字符串字面量
- 编译时检查，避免拼写错误
- IDE 自动补全支持

### 4. 数据分离

- **用户偏好**：存储在 `User.preferences`，作为默认值
- **临时过滤器**：`RecipeGenerationFilter`，用于单次生成
- **厨具**：从 `HouseholdUtensil` 获取，不是用户偏好
- **不喜欢的食材**：应从 `User.taboos` 获取，不是用户偏好

### 5. 验证机制

- 前端：使用 FilterChip 限制为标准库选项
- 后端：`UserService.updateUserPreferencesMap()` 验证所有输入值
- 拒绝非标准库的值，返回错误提示
