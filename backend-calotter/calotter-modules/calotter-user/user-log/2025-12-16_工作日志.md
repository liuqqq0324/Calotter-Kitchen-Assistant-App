## 🔄 运行逻辑

### 用户注册流程

1. **前端**：用户填写注册信息（username, email, password, confirmPassword）
2. **前端验证**：检查密码是否匹配（本地验证）
3. **前端请求**：`POST /api/user/register`（不发送 `confirmPassword`）
4. **后端处理**：
   - 检查用户名和邮箱是否已存在
   - 创建新用户（密码加密）
   - **自动创建默认家庭**（`{username}'s Home`）
   - 生成 JWT token
   - 返回 `AuthResponse`（包含 `token`、`userId`、`householdId`）
5. **前端保存**：将 `token`、`userId`、`householdId` 保存到 `SharedPreferences`
6. **前端跳转**：自动登录，跳转到主页面

### 用户登录流程

1. **前端**：用户输入用户名/邮箱和密码
2. **前端请求**：`POST /api/user/login`（`usernameOrEmail`、`password`）
3. **后端处理**：
   - 查找用户（支持用户名或邮箱）
   - 验证密码
   - 检查用户状态（未激活/封禁）
   - 生成 JWT token
   - **获取用户的第一个家庭**（如果有）
   - 返回 `AuthResponse`（包含 `token`、`userId`、`householdId`）
4. **前端保存**：将 `token`、`userId`、`householdId` 保存到 `SharedPreferences`
5. **前端跳转**：跳转到主页面

### 用户登出流程

1. **前端**：用户点击登出按钮
2. **前端请求**：`POST /api/user/logout`（带 Authorization header）
3. **后端处理**：
   - 验证 token（可选，主要用于日志）
   - 返回成功响应（JWT 无状态，无法主动失效）
4. **前端清除**：清除本地存储（`token`、`userId`、`householdId`）
5. **前端跳转**：返回登录页面

### 添加库存流程

1. **前端**：用户输入食材名称
2. **前端查找**：调用 `findStandardIngredientIdByName(name)` 查找标准食材ID
3. **前端验证**：如果找不到，显示错误提示
4. **前端请求**：`POST /api/inventory/ingredients`（包含 `householdId`、`standardIngredientId`、`quantity`、`unit`、`expirationDate`）
5. **后端处理**：
   - 验证家庭是否存在
   - 验证标准食材是否存在
   - 创建库存记录
   - 返回 `IngredientResponse`
6. **前端更新**：更新库存列表

## 📊 技术要点总结

### 1. Household 自动创建机制

- **设计决策**：个人用户 = 一个用户 + 一个默认家庭
- **实现方式**：注册时自动创建，登录时自动获取
- **优势**：对用户透明，架构支持未来扩展（多家庭成员）

### 2. JWT 无状态认证

- **登出处理**：JWT 无法主动失效，主要依赖前端清除本地存储
- **Token 验证**：后端可以验证 token 有效性，但无法主动失效
- **安全性**：依赖 token 过期时间，或使用黑名单机制（未实现）

### 3. 标准食材库设计

- **查找方式**：支持精确匹配和模糊匹配
- **数据关联**：库存必须关联标准食材库（保证数据一致性）
- **前端体验**：自动查找，如果找不到则提示用户

### 4. API 路径统一

- **后端路径**：`/api/user/*`、`/api/inventory/*`
- **前端调用**：部分使用 `/api/ums/user/*`（需要统一）
- **建议**：统一使用 `/api/user/*` 路径
