# Calotter Backend 项目总结文档

**更新时间：** 2025年12月16日 14:29  
**项目名称：** Personal Sous Chef Backend (Calotter)  
**技术栈：** Spring Boot 3.2.0 + JPA + PostgreSQL

---

## 1. 项目概述

Calotter 是一个基于 Spring Boot 的智能烹饪助手后端系统，采用模块化设计，支持用户管理、家庭管理、库存管理、烹饪会话管理和健康营养追踪等功能。系统通过 AI 生成个性化菜谱，并自动追踪营养摄入。

### 核心功能
- **用户认证与授权**：基于 JWT 的用户注册、登录
- **家庭管理**：支持多用户家庭组，邀请码机制
- **库存管理**：食材、厨具、调料、剩菜管理
- **AI 烹饪**：基于库存和用户偏好生成个性化菜谱
- **健康追踪**：自动记录营养摄入，生成健康报告

## 4. 模块详细说明

### 4.1 calotter-common（通用模块）

#### 核心类
- **BaseEntity**：所有实体的基类
  - 字段：`createTime`, `updateTime`, `createBy`, `updateBy`
  - 使用 `@MappedSuperclass` 和 JPA Auditing
  
- **Result<T>**：统一响应结果
  - 方法：`success()`, `error()`, `success(T data)`
  
- **PageResult<T>**：分页结果
  - 字段：`records`, `total`, `current`, `size`

#### 标准库实体
- **RefAllergen**：标准过敏原库
- **StandardIngredient**：标准食材库（包含营养信息、单位换算）
- **StandardUtensil**：标准厨具库
- **StandardSpice**：标准调料库

#### 配置类
- **JpaAuditingConfig**：JPA 审计配置（当前使用固定用户ID 1L）
- **GlobalExceptionHandler**：全局异常处理器

---

### 4.2 calotter-user（用户模块）

#### 实体类
- **User**：用户实体
  - 字段：`username`, `email`, `passwordHash`, `role`, `status`, `isOnboarded`
  - JSONB 字段：`settings`（用户设置）
  
- **Household**：家庭组实体
  - 字段：`name`, `inviteCode`, `ownerId`
  - 关联：`members`（家庭成员列表）
  
- **FamilyMember**：家庭成员实体
  - 字段：`name`, `gender`, `birthdate`, `currentHeight`, `currentWeight`
  - JSONB 字段：`dietaryStyles`, `preferences`（口味、菜系、不喜欢的食材）
  - 关联：`allergies`（过敏原列表）
  
- **HealthGoal**：健康目标实体
  - 字段：`goalType`（LOSE_FAT, MUSCLE_GAIN, MAINTENANCE）
  - 字段：`dailyCalories`, `protein`, `fat`, `carb`, `fiber`

#### 服务层
- **UserService**：用户注册、登录
- **HouseholdService**：家庭创建、更新、删除、邀请码管理
- **JwtService**：JWT Token 生成和验证

#### API 端点
- `POST /api/user/register` - 用户注册
- `POST /api/user/login` - 用户登录
- `POST /api/household` - 创建家庭
- `PUT /api/household/{id}` - 更新家庭
- `GET /api/household/{id}` - 获取家庭详情
- `GET /api/household/invite/{inviteCode}` - 通过邀请码获取家庭
- `GET /api/household/owner/{ownerId}` - 获取用户的所有家庭
- `DELETE /api/household/{id}` - 删除家庭

---

### 4.3 calotter-inventory（库存模块）

#### 实体类
- **Ingredient**：食材库存
  - 关联：`StandardIngredient`（标准食材）

- **HouseholdUtensil**：家庭厨具
  - 关联：`StandardUtensil`（标准厨具）
  - 字段：`isAvailable`, `remark`
  
- **HouseholdSpice**：家庭调料
  - 关联：`StandardSpice`（标准调料）
  - 字段：`isAvailable`, `remark`
  
- **LeftoverDish**：剩菜

#### 服务层
- **InventoryService**：库存管理服务
  - 食材、厨具、调料、剩菜的 CRUD 操作
  - 库存扣减功能

#### API 端点

- **调料管理**

  - `PUT /api/inventory/spices/{id}` - 更新调料
  - `GET /api/inventory/spices/{id}` - 获取调料详情
  - `GET /api/inventory/spices?householdId={id}` - 获取家庭所有调料
  - `DELETE /api/inventory/spices/{id}` - 删除调料

- **厨具管理**
  - `POST /api/inventory/utensils` - 创建厨具
  - `PUT /api/inventory/utensils/{id}` - 更新厨具
  - `GET /api/inventory/utensils/{id}` - 获取厨具详情
  - `GET /api/inventory/utensils?householdId={id}` - 获取家庭所有厨具

  - `POST /api/inventory/leftovers` - 创建剩菜
  - `PUT /api/inventory/leftovers/{id}` - 更新剩菜

### 4.4 calotter-cooking（烹饪模块）

#### 实体类
- **CookingSession**：烹饪会话
  - JSONB 字段：`requestContext`（AI 请求上下文）、`aiResponse`（AI 响应）
  - JSONB 字段：`ingredientsSnapshot`（用料快照）、`totalNutritionSnapshot`（营养快照）
  - 关联：`finalDish`（最终选择的 Dish）
  - 状态：`PENDING`, `COMPLETED`, `COOKED`, `CANCELLED`
  
- **Dish**：菜谱快照
  - 字段：`name`, `description`, `totalWeightGram`, `totalCalories`, `totalProtein`等
  - JSONB 字段：`steps`（烹饪步骤）、`tags`（标签）、`ingredientSnapshots`（原料快照）
  - 方法：`getCaloriesPer100g()`（计算每100克卡路里）

#### 服务层
- **CookingSessionService**：烹饪会话管理
  - `completeSession()`：完成烹饪会话，处理剩菜，发布事件
  
- **CookingWorkflowService**：烹饪工作流
  - `startCooking()`：开始烹饪
  - `finishCooking()`：结束烹饪
  
- **CookingContextBuilderService**：AI 上下文构建
  - `buildContext()`：构建 AI 烹饪上下文
  
- **DishService**：菜谱管理
  - `createDishFromAiResponse()`：从 AI 响应创建 Dish 快照
  
- **FavoriteRecipeService**：收藏菜谱管理
  
- **AiMenuService**：AI 菜单生成
  
- **LeftoverDishService**：剩菜服务（在 cooking 模块，但处理 inventory 的 LeftoverDish）

#### DTO 类
- **AiCookingContext**：AI 烹饪上下文
  - `TaskSettings`：任务设置
  - `CompositeNutritionalGoal`：复合营养目标
  - `DinerProfile`：食客画像
  - `KitchenSnapshot`：厨房快照
  
- **AiRecipeResponse**：AI 响应
  - `GeneratedDish`：生成的菜品
  - `PlatingInstruction`：分餐指南
  - `NutritionSummary`：营养总结

#### API 端点
- `POST /api/cooking/generate-context` - 生成 AI 烹饪上下文
- `POST /api/cooking/start` - 开始烹饪
- `POST /api/cooking/finish` - 结束烹饪
- `POST /api/cooking/complete` - 完成烹饪会话

---

### 4.5 calotter-health（健康模块）

#### 实体类
- **NutritionLog**：营养日志（摄入流水）
  - 字段：`logDate`, `eatenAt`, `mealType`, `sourceType`
  - 弱引用：`dishId`（关联 Dish）
  - 字段：`foodName`, `quantity`, `unit`
  - 营养字段：`calories`, `protein`, `fat`, `carb`, `fiber`
  - 来源类型：`APP_COOKING`, `LEFTOVER`, `MANUAL`, `EXTERNAL`
  
- **DailyNutrientAggregate**：每日营养聚合表
  - 用于快速查询每日营养汇总

#### 服务层
- **NutritionLogService**：营养日志服务
  - `createFromEvent()`：从烹饪完成事件创建日志
  - `createFromLeftover()`：从剩菜创建日志
  - `createManual()`：手动创建日志
  
- **NutritionAggregateService**：营养聚合服务
  - `getWeeklyReport()`：获取周健康报告
  
- **事件监听器**
  - `CookingSessionCompletedEventListener`：监听烹饪完成事件
  - `NutritionLogEventListener`：监听营养日志创建事件

#### API 端点
- `GET /api/nutrition/weekly` - 获取周健康报告
- `POST /api/nutrition/log/manual` - 手动记录营养摄入
- `POST /api/nutrition/log/leftover` - 从剩菜记录营养摄入

  - `HouseholdSpice`

## 6. 核心特性

### 6.1 JPA Auditing
- 所有实体继承 `BaseEntity`
- 自动填充 `createTime`, `updateTime`, `createBy`, `updateBy`
- 当前使用固定用户ID（1L），后续需集成 Spring Security

### 6.2 模块间解耦
- **弱引用设计**：`LeftoverDish.originalDishId`、`NutritionLog.dishId` 使用 Long 类型弱引用
- **事件驱动**：使用 Spring 事件机制解耦模块（如 `CookingSessionCompletedEvent`）
- **服务层查询**：通过 Service 层查询关联实体，避免直接引用

### 6.3 Copy-on-Write 策略
- **Dish 快照**：每次配方微调生成新的 Dish 记录，确保历史数据准确性
- **营养快照**：存储实际摄入的营养值，而非仅存储比例

### 6.4 事件驱动架构
- **CookingSessionCompletedEvent**：烹饪完成时发布，触发营养日志创建
- **NutritionLogCreatedEvent**：营养日志创建时发布，触发聚合表更新

### 6.5 异步处理
- **AsyncConfig**：配置异步任务执行器
- 营养日志创建和聚合表更新使用异步处理

## 9. 开发注意事项

### 9.1 模块依赖关系
- `calotter-common`：被所有模块依赖
- `calotter-inventory`：依赖 `calotter-user`
- `calotter-cooking`：依赖 `calotter-user`、`calotter-inventory`
- `calotter-health`：依赖 `calotter-user`、`calotter-cooking`（通过事件）
- `calotter-start`：聚合所有模块

### 9.2 数据库迁移
- 开发环境：使用 `ddl-auto: update` 自动更新表结构
- 生产环境：建议使用 Flyway 或 Liquibase 进行版本管理

### 9.3 安全配置
- 当前 SecurityConfig 允许所有请求
- 后续需要集成 JWT 过滤器进行身份验证

### 9.4 审计字段
- 当前 `createBy` 和 `updateBy` 使用固定值（1L）
- 后续需要从 Spring Security Context 获取当前用户ID
