## 项目概述

Personal Sous Chef 是一个智能个人厨师助手应用，帮助用户管理食材库存、生成个性化菜单、跟踪营养摄入。项目采用前后端分离架构。

### 技术栈

**后端：**
- Java 17
- Spring Boot 3.2.0
- Spring Data JPA
- PostgreSQL 15
- Maven 3.6+
- Lombok
- JWT (JSON Web Tokens)

**前端：**
- Flutter 3.10+
- Dart
- HTTP 客户端（`http` 包）
- SharedPreferences（本地存储）
- ONNX Runtime（图像识别）

**数据库：**
- PostgreSQL 15（生产环境）
- H2 Database（集成测试环境）

**容器化：**
- Docker Desktop
- Docker Compose

## 项目架构

### 1. 后端模块化架构

项目采用 Maven 多模块架构，实现关注点分离和模块复用：

```
backend-calotter/
├── pom.xml                          # 父工程（统一依赖版本管理）
├── calotter-common/                 # 通用模块
│   ├── core/                        # Result, PageResult 统一响应
│   ├── domain/                      # BaseEntity, 标准库实体
│   ├── exception/                   # 全局异常处理
│   └── config/                      # JPA Auditing 配置
├── calotter-modules/                # 业务模块聚合层
│   ├── calotter-user/               # 用户模块
│   │   ├── domain/entity/           # User, Household, HealthGoal
│   │   ├── repository/              # 数据访问接口
│   │   ├── service/                 # 业务服务层
│   │   └── controller/              # API 控制器
│   ├── calotter-health/             # 健康模块
│   │   ├── domain/entity/           # NutritionLog, DailyNutrientAggregate
│   │   ├── repository/              # 数据访问接口
│   │   ├── service/                 # 业务服务层
│   │   └── controller/              # API 控制器
│   ├── calotter-inventory/          # 库存模块
│   │   ├── domain/entity/           # Ingredient, HouseholdUtensil, HouseholdSpice, LeftoverDish
│   │   ├── repository/              # 数据访问接口
│   │   ├── service/                 # 业务服务层
│   │   └── controller/              # API 控制器
│   └── calotter-cooking/            # 烹饪模块
│       ├── domain/entity/           # CookingSession, Dish
│       ├── repository/              # 数据访问接口
│       ├── service/                 # 业务服务层
│       └── controller/              # API 控制器
└── calotter-start/                  # 启动模块
    ├── CalotterApplication.java     # 启动类
    └── application.yml              # 配置文件
```

### 2. 模块依赖关系

```
calotter-common (基础模块)
    ↑
    ├── calotter-user (用户模块)
    │       ↑
    │       ├── calotter-inventory (库存模块)
    │       │       ↑
    │       │       └── calotter-cooking (烹饪模块)
    │       │
    │       └── calotter-health (健康模块)
    │
    └── calotter-start (启动模块，聚合所有模块)
```

**依赖规则：**
- 所有模块依赖 `calotter-common`
- `calotter-inventory` 依赖 `calotter-user`
- `calotter-cooking` 依赖 `calotter-user` 和 `calotter-inventory`
- `calotter-health` 依赖 `calotter-user`
- `calotter-start` 聚合所有业务模块

## 数据结构及关系

### 1. 核心实体关系图

```
User (用户)
  ├── Household (家庭组) [1:N]
  │   ├── Ingredient (食材库存) [1:N]
  │   ├── HouseholdUtensil (厨具) [1:N]
  │   ├── HouseholdSpice (调料) [1:N]
  │   └── LeftoverDish (剩菜) [1:N]
  │
  ├── HealthGoal (健康目标) [1:1]
  ├── NutritionLog (营养日志) [1:N]
  └── DailyNutrientAggregate (每日营养汇总) [1:N]

CookingSession (烹饪会话)
  ├── Dish (菜品) [1:N]
  └── LeftoverDish (剩菜) [1:N]

StandardIngredient (标准食材库)
  └── Ingredient (食材库存) [N:1]

StandardUtensil (标准厨具库)
  └── HouseholdUtensil (厨具) [N:1]

StandardSpice (标准调料库)
  └── HouseholdSpice (调料) [N:1]

RefAllergen (标准过敏原库)
  ├── StandardIngredient (标准食材) [N:M]
  └── StandardSpice (标准调料) [N:M]
```

### 2. 主要实体说明

#### User（用户）
- `id`: 主键
- `username`: 用户名（唯一）
- `email`: 邮箱（唯一）
- `passwordHash`: 密码哈希
- `role`: 角色（ROLE_USER, ROLE_ADMIN）
- `status`: 状态（1=激活，0=禁用）
- `isOnboarded`: 是否完成引导
- `settings`: JSONB 字段，存储用户设置

#### Household（家庭组）
- `id`: 主键
- `name`: 家庭组名称
- `inviteCode`: 邀请码（唯一）
- `ownerId`: 所有者用户ID（外键关联 User）

#### Ingredient（食材库存）
- `id`: 主键
- `householdId`: 家庭组ID（外键关联 Household）
- `standardIngredientId`: 标准食材ID（外键关联 StandardIngredient）
- `quantity`: 数量
- `unit`: 单位（如 "g", "kg", "pcs"）
- `expirationDate`: 过期日期
- `location`: 存储位置（FRIDGE, FREEZER, PANTRY）

#### StandardIngredient（标准食材库）
- `id`: 主键（手动分配，如 1001）
- `name`: 食材名称（英文）
- `category`: 分类（MEAT, VEG, FRUIT, GRAIN, DAIRY, NUT, OTHER）
- `calories`: 卡路里（每100g）
- `protein`: 蛋白质（每100g）
- `fat`: 脂肪（每100g）
- `carb`: 碳水化合物（每100g）
- `fiber`: 纤维（每100g）
- `averageGramPerUnit`: 单位换算（每单位对应的克数）
- `shelfLifePantry/Fridge/Freezer`: 保质期（天）
- `defaultLocation`: 默认存储位置

#### CookingSession（烹饪会话）
- `id`: 主键
- `householdId`: 家庭组ID
- `status`: 状态（IN_PROGRESS, COMPLETED, CANCELLED）
- `requestContext`: JSONB 字段，存储请求上下文
- `aiResponse`: JSONB 字段，存储 AI 响应

#### NutritionLog（营养日志）
- `id`: 主键
- `userId`: 用户ID（外键关联 User）
- `source`: 来源（RECIPE, MANUAL, LEFTOVER）
- `consumedPercentage`: 消费百分比（0-100）
- `nutrition`: JSONB 字段，存储营养信息

## API 接口文档

### 统一响应格式

所有 API 响应遵循统一格式：

```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

**状态码说明：**
- `200`: 成功
- `400`: 请求参数错误
- `401`: 未授权
- `403`: 禁止访问
- `404`: 资源不存在
- `500`: 服务器内部错误

### 1. 用户模块 API

#### 1.1 用户注册
```
POST /api/user/register
POST /api/ums/user/register  # 兼容路径
```

**请求体：**
```json
{
  "username": "testuser",
  "email": "test@example.com",
  "password": "password123"
}
```

**响应：**
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
```

## 环境部署指南

### 1. 环境要求

**必需软件：**
- Java 17+
- Maven 3.6+
- Docker Desktop
- PostgreSQL 15（或使用 Docker）
- Flutter 3.10+（前端开发）

**可选工具：**
- IntelliJ IDEA / Eclipse（IDE）
- Postman（API 测试）
- pgAdmin（数据库管理）

### 2. 新环境部署（纯净环境）

#### 2.1 环境检查

**检查 Java 版本：**
```bash
java -version
# 应显示：openjdk version "17" 或更高
```

**检查 Maven 版本：**
```bash
mvn -version
# 应显示：Apache Maven 3.6.x 或更高
```

**检查 Docker Desktop：**
- 打开 Docker Desktop
- 确认 Docker 服务正在运行
- 在终端执行：`docker ps`，应能正常显示容器列表

#### 2.2 数据库准备

**启动 PostgreSQL 容器：**
```bash
# 进入项目根目录
cd backend-calotter

# 启动数据库容器
docker-compose up -d postgres

# 验证容器运行状态
docker ps
# 应看到 calotter_postgres 容器正在运行
```

**验证数据库连接：**
```bash
# 连接数据库
docker exec -it calotter_postgres psql -U postgres -d calotter

# 在 PostgreSQL 命令行中执行
\dt
# 应显示表列表（首次启动时可能为空）

# 退出
\q
```

**执行数据库迁移（如果表已存在）：**
```bash
# 如果数据库已有表结构，需要执行迁移脚本添加新字段
# Windows PowerShell
Get-Content backend-calotter/calotter-modules/calotter-inventory/migration/add-leftover-snapshot-fields.sql | docker exec -i calotter_postgres psql -U postgres -d calotter

# macOS/Linux
cat backend-calotter/calotter-modules/calotter-inventory/migration/add-leftover-snapshot-fields.sql | docker exec -i calotter_postgres psql -U postgres -d calotter

# 验证迁移结果
docker exec -it calotter_postgres psql -U postgres -d calotter -c "\d household_leftovers"
# 应看到 dish_name, cover_image, calories_per_100g 字段
```

**初始化标准库数据：**
```bash
# Windows PowerShell
Get-Content backend-calotter/init-standard-libraries.sql | docker exec -i calotter_postgres psql -U postgres -d calotter

# macOS/Linux
cat backend-calotter/init-standard-libraries.sql | docker exec -i calotter_postgres psql -U postgres -d calotter
```

**验证数据初始化：**
```bash
docker exec -it calotter_postgres psql -U postgres -d calotter -c "SELECT COUNT(*) FROM ref_standard_ingredients;"
# 应返回：77

docker exec -it calotter_postgres psql -U postgres -d calotter -c "SELECT COUNT(*) FROM ref_standard_spices;"
# 应返回：30

docker exec -it calotter_postgres psql -U postgres -d calotter -c "SELECT COUNT(*) FROM ref_standard_utensils;"
# 应返回：25
```

#### 2.3 后端编译和启动

**编译项目：**

#### 2.4 AI API 配置（重要）

**⚠️ 重要提示：**
- **开发测试阶段必须配置自己的 API KEY**，否则会遇到 `429 Too Many Requests`（Token 配额超出）错误
- 默认使用 **Mock 模式**（不需要 API KEY），适合前端 UI 开发和测试
- 如需测试真实的 AI 菜单生成功能，请配置自己的 API KEY

**支持的 AI 提供商：**
- **Mock**（默认）- 返回固定假数据，0 费用、0 延迟
- **Gemini**（优化版）- 简化 Prompt，Token 消耗降低 50%-70%
- **Groq**（完整版）- 保留原有完整 Prompt 逻辑

**配置步骤：**

**步骤1：选择模式**

编辑 `calotter-start/src/main/resources/application.yml`：

```yaml
ai:
  api:
    mode: mock  # mock（默认）/ gemini（优化版）/ groq（完整版）
```

**模式说明：**
- **`mock`**（默认）：开发测试阶段使用，返回固定假数据
  - ✅ 不需要 API Key
  - ✅ 0 费用、0 延迟
  - ✅ 前端可以无限制测试 UI
- **`gemini`**：使用 Gemini API，优化版（简化 Prompt）
  - ✅ Token 消耗降低 50%-70%
  - ✅ 适合免费版配额有限的情况
- **`groq`**：使用 Groq API，完整 Prompt 版本
  - ✅ 保留原有完整逻辑

**步骤2：配置 API Key（如果使用 gemini 或 groq 模式）**

**方法1：使用环境变量（推荐）**

**Windows PowerShell:**
```powershell
# 设置 Gemini API Key
$env:GEMINI_API_KEY = "your_gemini_api_key_here"

# 设置 Groq API Key
$env:GROQ_API_KEY = "your_groq_api_key_here"
```

**macOS/Linux:**
```bash
# 设置 Gemini API Key
export GEMINI_API_KEY="your_gemini_api_key_here"

# 设置 Groq API Key
export GROQ_API_KEY="your_groq_api_key_here"
```

**方法2：直接在配置文件中设置（不推荐，仅用于测试）**

编辑 `calotter-start/src/main/resources/application.yml`：

```yaml
ai:
  api:
    mode: gemini  # 或 groq
    gemini:
      api-key: "your_gemini_api_key_here"  # ⚠️ 不要提交到 Git
      base-url: https://generativelanguage.googleapis.com
      model: gemini-2.5-pro
    groq:
      api-key: "your_groq_api_key_here"  # ⚠️ 不要提交到 Git
      url: https://api.groq.com/openai/v1/chat/completions
      model: llama-3.3-70b-versatile
```

**步骤3：获取 API Key**

**Gemini API Key：**
1. 访问 [Google AI Studio](https://makersuite.google.com/app/apikey)
2. 登录 Google 账号
3. 创建新的 API Key
4. 复制 API Key 并配置到环境变量或配置文件中

**Groq API Key：**
1. 访问 [Groq Console](https://console.groq.com/)
2. 注册/登录账号
3. 创建新的 API Key
4. 复制 API Key 并配置到环境变量或配置文件中

**步骤4：重启后端服务**

配置完成后，需要重启后端服务使配置生效：

```bash
# 停止当前运行的后端服务（Ctrl+C）
# 然后重新启动
cd calotter-start
mvn spring-boot:run
```

**验证配置：**
- 查看后端启动日志，确认使用的模式（Mock/Gemini/Groq）
- 测试菜单生成 API：
  ```bash
  # Windows PowerShell
  Invoke-RestMethod -Uri "http://localhost:8080/api/ai/generate-menus" -Method Post -ContentType "application/json" -Body '{"householdId":1}'
  
  # macOS/Linux
  curl -X POST http://localhost:8080/api/ai/generate-menus \
    -H "Content-Type: application/json" \
    -d '{"householdId":1}'
  ```

**常见问题：**

**问题1：429 Too Many Requests（Token 配额超出）**
- **原因**：使用了共享的 API Key，配额已用完
- **解决方案**：
  1. **立即切换到 Mock 模式**（推荐用于开发测试）：
     ```yaml
     ai:
       api:
         mode: mock
     ```
  2. 配置并使用自己的 API Key
  3. 等待配额重置（通常是按天或按月）
  4. 使用 Gemini 优化版（`mode: gemini`）可以降低 Token 消耗

**问题2：API Key 未配置错误**
- **解决方案**：
  1. 检查环境变量是否设置：
     ```bash
     # Windows PowerShell
     echo $env:GEMINI_API_KEY
     
     # macOS/Linux
     echo $GEMINI_API_KEY
     ```
  2. 检查 `application.yml` 中的配置
  3. 重启应用

**问题3：切换模式后不生效**
- **解决方案**：
  1. 确认配置文件已保存
  2. 重启应用
  3. 检查日志确认使用的模式

**最佳实践：**
1. **开发测试阶段使用 Mock 模式**：不需要 API Key，可以无限制测试 UI
2. **使用环境变量配置 API Key**：避免将敏感信息提交到 Git
3. **每个开发者使用自己的 API Key**：避免共享配额导致 429 错误
4. **API Key 安全**：不要将 API Key 提交到 Git 仓库，使用环境变量或 `application-local.yml`（已添加到 `.gitignore`）

**详细配置说明：**
- 更多配置选项和故障排查，请参考 `backend-calotter/AI_API_CONFIG.md`

### 5. 常见问题排查

#### 4.1 Docker 相关问题

**问题：** Docker Desktop 无法启动
- **解决：** 检查 Docker Desktop 是否已安装并启动，重启 Docker Desktop

**问题：** 容器无法启动
- **解决：** 
  ```bash
  # 查看容器日志
  docker logs calotter_postgres
  
  # 重启容器
  docker-compose restart postgres
  ```

#### 5.2 数据库连接问题

**问题：** 无法连接到数据库
- **检查：** 
  1. 确认 PostgreSQL 容器正在运行：`docker ps`
  2. 检查 `application.yml` 中的数据库配置
  3. 确认端口 5432 未被占用

**问题：** 表不存在
- **解决：** 
  1. 确认 `ddl-auto: update` 配置正确
  2. 重启后端服务，JPA 会自动创建表

#### 5.3 Maven 编译问题

**问题：** 编译失败
- **解决：**
  ```bash
  # 清理 Maven 缓存
  mvn clean
  
  # 重新编译
  mvn compile -DskipTests
  ```

**问题：** 依赖下载失败
- **解决：**
  1. 检查网络连接
  2. 配置 Maven 镜像（如阿里云镜像）

#### 5.4 端口冲突

**问题：** 8080 端口被占用
- **解决：**
  1. 修改 `application.yml` 中的 `server.port`
  2. 或停止占用 8080 端口的其他服务

```powershell
# 启动数据库
```
