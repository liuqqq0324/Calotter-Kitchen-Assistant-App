## 项目概述

Personal Sous Chef 是一个智能个人厨师助手应用，帮助用户管理食材库存、生成个性化菜单、跟踪营养摄入。项目采用前后端分离架构，后端使用 Spring Boot + JPA，前端使用 Flutter。

### 技术栈

**后端：**
- Java 17
- Spring Boot 3.2.0
- Spring Data JPA
- PostgreSQL
- Maven
- Lombok

**前端：**
- Flutter
- Dart
- HTTP 客户端（`http` 包）
- SharedPreferences（本地存储）

## 后端架构

### 1. 模块化设计

项目采用 Maven 多模块架构，实现关注点分离和模块复用：

```
backend-calotter/
├── pom.xml                          # 父工程
├── calotter-common/                 # 通用模块
│   ├── core/                        # Result, PageResult, 异常处理
│   ├── domain/                      # BaseEntity, 标准库实体
│   └── config/                      # JPA Auditing 配置
├── calotter-modules/                # 业务模块聚合层
│   ├── calotter-user/               # 用户模块
│   ├── calotter-health/             # 健康模块
│   ├── calotter-inventory/          # 库存模块
│   └── calotter-cooking/            # 烹饪模块
└── calotter-start/                  # 启动模块
    ├── CalotterApplication.java     # 启动类
    └── application.yml              # 配置文件
```

### 2. 模块依赖关系

```
calotter-common (基础模块)
    ↑
    ├── calotter-user (用户模块)
    │       ↑
    │       ├── calotter-inventory (库存模块)
    │       │       ↑
    │       │       └── calotter-cooking (烹饪模块)
    │       │
    │       └── calotter-health (健康模块)
    │
    └── calotter-start (启动模块，聚合所有模块)
```

**依赖规则：**
- 所有模块依赖 `calotter-common`
- `calotter-inventory` 依赖 `calotter-user`
- `calotter-cooking` 依赖 `calotter-user` 和 `calotter-inventory`
- `calotter-health` 依赖 `calotter-user`
- `calotter-start` 聚合所有业务模块

### 3. 分层架构

每个业务模块遵循标准分层架构：

```
模块结构/
├── domain/entity/          # 实体类（Entity）
├── repository/             # 数据访问层（Repository）
├── service/                # 业务逻辑层（Service）
└── controller/             # 控制器层（Controller）
```

**各层职责：**

- **Entity（实体层）**：JPA 实体类，映射数据库表
- **Repository（数据访问层）**：继承 `JpaRepository`，提供数据访问方法
- **Service（业务逻辑层）**：实现业务逻辑，调用 Repository
- **Controller（控制器层）**：处理 HTTP 请求，调用 Service，返回统一响应格式

### 4. 核心特性

#### 4.1 JPA Auditing

所有实体继承 `BaseEntity`，自动记录创建时间、更新时间、创建人、更新人：

```java
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {
    @CreatedDate
    private LocalDateTime createTime;
    
    @LastModifiedDate
    private LocalDateTime updateTime;
    
    @CreatedBy
    private Long createBy;
    
    @LastModifiedBy
    private Long updateBy;
}
```

#### 4.2 JSONB 支持

使用 PostgreSQL JSONB 存储复杂数据结构：

```java
@JdbcTypeCode(SqlTypes.JSON)
@Column(columnDefinition = "jsonb")
private Map<String, Object> settings;
```

#### 4.3 统一响应格式

所有 API 返回统一的 `Result<T>` 格式：

```java
{
    "code": 200,
    "message": "操作成功",
    "data": { ... }
}
```

## 数据结构及关系

### 1. 核心实体关系图

```
User (用户)
├── HealthGoal (健康目标) [1:N]
├── Household (家庭组) [N:M]
│   ├── Ingredient (食材库存) [1:N]
│   ├── HouseholdUtensil (家庭厨具) [1:N]
│   ├── HouseholdSpice (家庭调料) [1:N]
│   └── LeftoverDish (剩菜) [1:N]
│
├── CookingSession (烹饪会话) [N:1]
│   └── Dish (菜品快照) [N:M]
│
└── NutritionLog (营养日志) [1:N]
    └── DailyNutrientAggregate (每日营养聚合) [1:N]

标准库实体（独立）:
├── StandardIngredient (标准食材)
├── StandardUtensil (标准厨具)
├── StandardSpice (标准调料)
└── RefAllergen (标准过敏原)
```

### 2. 实体详细说明

#### 2.1 用户模块实体

**User（用户）**
- `id` - 用户ID（主键）
- `username` - 用户名（唯一）
- `email` - 邮箱（唯一）
- `passwordHash` - 密码哈希
- `role` - 角色（ROLE_USER, ROLE_ADMIN）
- `status` - 状态（0:未激活, 1:可用, 2:封禁）
- `isOnboarded` - 是否完成引导
- `currentHouseholdId` - 当前活跃家庭ID
- `allergies` - 过敏原列表（关联 RefAllergen）
- `preferences` - 偏好设置（JSONB）
- `settings` - 用户设置（JSONB）

**Household（家庭组）**
- `id` - 家庭ID（主键）
- `name` - 家庭名称
- `inviteCode` - 邀请码（唯一）
- `ownerId` - 所有者ID（关联 User）
- `members` - 成员列表（ManyToMany 关联 User）

**HealthGoal（健康目标）**
- `id` - 目标ID（主键）
- `user` - 用户（ManyToOne 关联 User）
- `status` - 状态（0:未激活, 1:激活）
- `dailyCalories` - 每日卡路里目标
- `goalType` - 目标类型（fat_loss, muscle_gain, maintain）

#### 2.2 库存模块实体

**Ingredient（食材库存）**
- `id` - 食材ID（主键）
- `household` - 家庭（ManyToOne 关联 Household）
- `metadata` - 标准食材（ManyToOne 关联 StandardIngredient）
- `quantity` - 数量
- `unit` - 单位（g, ml, pcs）
- `expirationDate` - 过期日期
- `location` - 存放位置（FRIDGE, FREEZER, PANTRY）

**HouseholdUtensil（家庭厨具）**
- `id` - 厨具ID（主键）
- `household` - 家庭（ManyToOne 关联 Household）
- `metadata` - 标准厨具（ManyToOne 关联 StandardUtensil）
- `isAvailable` - 是否可用

**HouseholdSpice（家庭调料）**
- `id` - 调料ID（主键）
- `household` - 家庭（ManyToOne 关联 Household）
- `metadata` - 标准调料（ManyToOne 关联 StandardSpice）
- `isAvailable` - 是否可用

**LeftoverDish（剩菜）**
- `id` - 剩菜ID（主键）
- `household` - 家庭（ManyToOne 关联 Household）
- `originalDishId` - 原始菜品ID（关联 Dish）
- `currentQuantityGram` - 当前剩余重量（克）
- `producedTime` - 制作时间

#### 2.3 烹饪模块实体

**CookingSession（烹饪会话）**
- `id` - 会话ID（主键）
- `householdId` - 家庭ID
- `initiatorId` - 发起人ID（关联 User）
- `menuId` - 菜单ID（1-5）
- `status` - 状态（PENDING, COOKING, COOKED）
- `dishes` - 菜品列表（OneToMany 关联 Dish）
- `finalDish` - 主菜（ManyToOne 关联 Dish，向后兼容）
- `remainingRatio` - 剩余比例
- `ingredientsSnapshot` - 用料快照（JSONB）
- `totalNutritionSnapshot` - 营养快照（JSONB）

**Dish（菜品快照）**
- `id` - 菜品ID（主键）
- `name` - 菜品名称
- `description` - 描述
- `totalCalories` - 总卡路里
- `totalProtein` - 总蛋白质（克）
- `totalFat` - 总脂肪（克）
- `totalCarb` - 总碳水化合物（克）
- `totalWeightGram` - 总重量（克）
- `difficulty` - 难度（EASY, MEDIUM, HARD）
- `steps` - 制作步骤（JSONB）

#### 2.4 健康模块实体

**NutritionLog（营养日志）**
- `id` - 日志ID（主键）
- `user` - 用户（ManyToOne 关联 User）
- `logDate` - 记录日期
- `sourceType` - 来源类型（APP_COOKING, MANUAL）
- `foodName` - 食物名称
- `dishId` - 菜品ID（关联 Dish，可选）
- `energy` - 能量（卡路里）
- `protein` - 蛋白质（克）
- `fat` - 脂肪（克）
- `carbohydrates` - 碳水化合物（克）
- `consumedPercentage` - 消费百分比（0-100）
- `baseEnergy` - 基础能量（100%时的值）
- `baseProtein` - 基础蛋白质
- `baseFat` - 基础脂肪
- `baseCarbohydrates` - 基础碳水化合物

**DailyNutrientAggregate（每日营养聚合）**
- `id` - 聚合ID（主键）
- `user` - 用户（ManyToOne 关联 User）
- `date` - 日期
- `totalEnergy` - 总能量
- `totalProtein` - 总蛋白质
- `totalFat` - 总脂肪
- `totalCarbohydrates` - 总碳水化合物
- `goalEnergySnapshot` - 目标能量快照
- `status` - 状态（ACTIVE, ARCHIVED）

#### 2.5 标准库实体

**StandardIngredient（标准食材）**
- `id` - 食材ID（主键，手动分配）
- `name` - 食材名称
- `category` - 类别（MEAT, VEG, FRUIT等）
- `calories` - 卡路里（每100g）
- `protein` - 蛋白质（每100g）
- `fat` - 脂肪（每100g）
- `carb` - 碳水化合物（每100g）
- `averageGramPerUnit` - 单位换算（1单位=多少克）
- `shelfLifePantry` - 常温保质期（天）
- `shelfLifeFridge` - 冷藏保质期（天）
- `shelfLifeFreezer` - 冷冻保质期（天）
- `defaultLocation` - 默认存放位置

**StandardUtensil（标准厨具）**
- `id` - 厨具ID（主键，手动分配）
- `name` - 厨具名称
- `iconUrl` - 图标URL

**StandardSpice（标准调料）**
- `id` - 调料ID（主键，手动分配）
- `name` - 调料名称
- `containedAllergens` - 包含的过敏原（ManyToMany 关联 RefAllergen）

**RefAllergen（标准过敏原）**
- `id` - 过敏原ID（主键）
- `name` - 过敏原名称
- `description` - 描述

### 3. 关键关系说明

#### 3.1 User 与 Household 关系

- **ManyToMany**：一个用户可以加入多个家庭，一个家庭可以有多个成员
- **关联表**：`user_household`（JPA自动创建）
- **拥有方**：User（`joinedHouseholds` 字段）

#### 3.2 Household 与库存实体关系

- **OneToMany**：一个家庭可以有多个食材、厨具、调料、剩菜
- **级联删除**：删除家庭时，自动删除所有关联的库存数据（通过数据库外键）

#### 3.3 CookingSession 与 Dish 关系

- **OneToMany**：一个会话可以包含多道菜（支持菜单）
- **向后兼容**：保留 `finalDish` 字段，支持单道菜场景

#### 3.4 User 与 NutritionLog 关系

- **OneToMany**：一个用户可以有多个营养日志
- **聚合表**：`DailyNutrientAggregate` 用于快速查询每日营养汇总

## 前后端通讯

### 1. API 基础配置

**后端基础URL：**
```
http://localhost:8080
```

**前端API配置：**
```dart
// lib/services/api_config.dart
class ApiConfig {
  static const String baseUrl = 'http://localhost:8080';
  static const String apiPrefix = '/api';
}
```

### 2. 认证机制

**JWT Token 认证：**
- 登录成功后，后端返回 JWT Token
- 前端保存 Token 到 SharedPreferences
- 后续请求在 Header 中携带 Token：
  ```
  Authorization: Bearer <token>
  ```

**Token 有效期：**
- 默认：3000秒（50分钟）
- 配置位置：`application.yml` → `jwt.expiration`

### 3. 统一响应格式

**成功响应：**
```json
{
  "code": 200,
  "message": "操作成功",
  "data": { ... }
}
```

**错误响应：**
```json
{
  "code": 400,
  "message": "错误描述",
  "data": null
}
```

### 4. 主要API接口

#### 4.1 用户认证API

**注册：**
```
POST /api/user/register
Content-Type: application/json

{
  "username": "testuser",
  "password": "password123",
  "email": "test@example.com"
}
```

**登录：**
```
POST /api/user/login
Content-Type: application/json

{
  "usernameOrEmail": "testuser",
  "password": "password123"
}
```

#### 4.2 营养相关API

**获取周营养目标：**
```
GET /api/nutrition/targets/weekly?userId={userId}&weekStart={yyyy-MM-dd}
Authorization: Bearer <token>
```

**获取周营养摘要：**
```
GET /api/nutrition/summary?userId={userId}&period=week
Authorization: Bearer <token>
```

**获取今日摄入：**
```
GET /api/intake/today?userId={userId}&source={recipe|manual|all}
Authorization: Bearer <token>
```

**添加手动摄入：**
```
POST /api/intake/manual?userId={userId}
Content-Type: application/json
Authorization: Bearer <token>

{
  "date": "2025-01-15",
  "foodName": "fried rice with egg",
  "portionDescription": "1 bowl"
}
```

**更新摄入百分比：**
```
PATCH /api/intake/{intakeId}?userId={userId}
Content-Type: application/json
Authorization: Bearer <token>

{
  "consumedPercentage": 80.0
}
```

#### 4.3 食谱相关API

**获取默认Filter：**
```
GET /api/menu/default-filter?householdId={householdId}
Authorization: Bearer <token>
```

**生成菜单：**
```
POST /api/menu/generate?householdId={householdId}
Content-Type: application/json
Authorization: Bearer <token>

{
  "dietPreferences": { ... },
  "calorieTarget": { ... },
  "servings": 2,
  ...
}
```

#### 4.4 烹饪相关API

**开始烹饪：**
```
POST /api/cooking/start
Content-Type: application/json
Authorization: Bearer <token>

{
  "householdId": 1,
  "initiatorId": 1,
  "menuId": 1,
  "recipes": [ ... ]
}
```

**完成烹饪：**
```
POST /api/cooking/finish
Content-Type: application/json
Authorization: Bearer <token>

{
  "sessionId": 1,
  "completedDishIds": [1, 2],
  "finalIngredients": [ ... ],
  "totalNutrition": { ... },
  "diners": [ ... ]
}
```

#### 4.5 库存相关API

**获取标准库：**
```
GET /api/inventory/standard-ingredients
GET /api/inventory/standard-utensils
GET /api/inventory/standard-spices
GET /api/user/standard-allergens
```

**食材CRUD：**
```
POST /api/inventory/ingredients
GET /api/inventory/ingredients?householdId={householdId}
PUT /api/inventory/ingredients/{id}
DELETE /api/inventory/ingredients/{id}
```

**切换厨具/调料可用性：**
```
PATCH /api/inventory/utensils/{id}/toggle
PATCH /api/inventory/spices/{id}/toggle
```

### 5. 数据格式规范

**命名规范：**
- 统一使用 **camelCase**（驼峰命名）
- 前端和后端保持一致

**日期格式：**
- 请求：`yyyy-MM-dd`（如：`2025-01-15`）
- 响应：ISO 8601 格式（如：`2025-01-15T00:00:00`）

**数字格式：**
- 能量：整数（卡路里）
- 营养值：小数（克），保留2位小数
- 百分比：小数（0-100），保留2位小数

### 6. 常见问题解决

#### 6.1 后端问题

**问题：数据库连接失败**
```
解决方案：
1. 检查 PostgreSQL 是否运行：docker ps 或 ps aux | grep postgres
2. 检查连接配置：application.yml
3. 检查数据库用户权限
4. 检查防火墙设置
```

**问题：端口被占用**
```
解决方案：
1. 查找占用进程：netstat -ano | findstr :8080 (Windows)
2. 或：lsof -i :8080 (macOS/Linux)
3. 修改 application.yml 中的 server.port
```

**问题：依赖下载失败**
```
解决方案：
1. 检查网络连接
2. 配置 Maven 镜像（settings.xml）
3. 清理并重新下载：mvn clean install -U
```

#### 6.2 前端问题

**问题：无法连接后端**
```
解决方案：
1. 检查 ApiConfig.baseUrl 配置
2. 检查后端服务是否运行
3. 检查网络连接
4. Android模拟器使用 10.0.2.2:8080
5. iOS模拟器使用 localhost:8080
6. 真机使用实际IP地址
```

**问题：Token 丢失**
```
解决方案：
1. 检查 SharedPreferences 权限
2. 检查 Token 保存逻辑
3. 检查 Token 过期时间
```

**问题：标准库加载失败**
```
解决方案：
1. 检查网络请求
2. 检查后端标准库API是否正常
3. 检查缓存逻辑
4. 清除缓存重新加载
```
