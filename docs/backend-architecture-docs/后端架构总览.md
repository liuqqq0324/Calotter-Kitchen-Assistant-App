## 项目概述

Calotter 是一个**智能烹饪助手后端系统**，采用 **Spring Boot** 框架和 **Maven 多模块**架构。整个系统被设计成多个独立的模块，每个模块负责不同的业务功能，便于团队协作开发和维护。

### 核心功能

- 👤 **用户管理**：用户注册、登录、家庭管理、家庭成员管理
- 📦 **库存管理**：食材、调料、厨具、剩菜的管理
- 🍳 **烹饪推荐**：基于库存和用户偏好生成AI烹饪建议
- 💊 **健康管理**：记录营养摄入、生成健康报告

### 模块说明

整个后端项目采用 **Maven 多模块**结构，分为以下几个部分：

1. **calotter-common**（通用模块）
   - 作用：提供所有模块共享的基础功能
   - 包含：统一响应格式、异常处理、基础实体类、标准库实体

2. **calotter-modules**（业务模块聚合层）
   - 作用：包含所有业务功能模块
   - 包含：user、inventory、cooking、health 四个业务模块

3. **calotter-start**（启动模块）
   - 作用：整个应用的唯一入口
   - 包含：主启动类、配置文件

## 模块划分

### 1. calotter-common（通用模块）

**位置**：`backend-calotter/calotter-common/`

**作用**：提供所有业务模块共享的基础功能，避免代码重复。

**包含内容**：

```
calotter-common/
├── core/                          # 核心类
│   ├── Result.java               # 统一响应格式（所有API返回都用这个）
│   ├── PageResult.java           # 分页响应格式
│   └── domain/
│       ├── BaseEntity.java       # 所有实体类的基类（包含创建时间、更新时间等）
│       └── entity/               # 标准库实体（字典表）
│           ├── StandardIngredient.java    # 标准食材库
│           ├── StandardUtensil.java       # 标准厨具库
│           ├── StandardSpice.java         # 标准调料库
│           └── RefAllergen.java           # 标准过敏原库
├── exception/                     # 异常处理
│   └── GlobalExceptionHandler.java  # 全局异常处理器（统一处理所有异常）
└── config/                        # 配置类
    └── JpaAuditingConfig.java     # JPA审计配置（自动填充创建时间等）
```

**为什么需要这个模块？**

想象一下，如果每个模块都自己写响应格式、异常处理，那代码会重复很多。`calotter-common` 就像一个"工具箱"，所有模块都可以使用里面的工具。

---

### 2. calotter-modules（业务模块聚合层）

**位置**：`backend-calotter/calotter-modules/`

**作用**：包含所有业务功能模块，每个模块负责一个独立的业务领域。

**包含模块**：

- **calotter-user**：用户模块
- **calotter-inventory**：库存模块
- **calotter-cooking**：烹饪模块
- **calotter-health**：健康模块

**为什么这样划分？**

每个模块可以独立开发和测试，不同的人可以负责不同的模块，减少代码冲突。

---

### 3. calotter-start（启动模块）

**位置**：`backend-calotter/calotter-start/`

**作用**：整个应用的唯一入口，负责启动整个Spring Boot应用。

**包含内容**：

```
calotter-start/
├── src/main/java/
│   └── com/calotter/
│       └── CalotterApplication.java  # 主启动类（包含main方法）
└── src/main/resources/
    └── application.yml              # 配置文件（数据库连接等）
```

**为什么需要这个模块？**

Spring Boot 应用需要一个入口点来启动。`calotter-start` 模块依赖所有业务模块，启动时会加载所有模块的功能。

## 分层架构

每个业务模块都遵循相同的分层架构，从外到内分为四层：

```
┌─────────────────────────────────────┐
│   Controller 层（控制器层）            │
│   作用：接收HTTP请求，返回响应         │
│   示例：UserController               │
└──────────────┬──────────────────────┘
               │ 调用
               ▼
┌─────────────────────────────────────┐
│   Service 层（业务逻辑层）             │
│   作用：处理业务逻辑，调用Repository   │
│   示例：UserService                  │
└──────────────┬──────────────────────┘
               │ 调用
               ▼
┌─────────────────────────────────────┐
│   Repository 层（数据访问层）          │
│   作用：操作数据库，增删改查           │
│   示例：UserRepository               │
└──────────────┬──────────────────────┘
               │ 操作
               ▼
┌─────────────────────────────────────┐
│   Entity 层（实体层）                  │
│   作用：定义数据库表结构               │
│   示例：User                         │
└─────────────────────────────────────┘
```

### 各层详细说明

#### 1. Controller 层（控制器层）

**作用**：接收前端发来的HTTP请求，调用Service层处理，返回结果。

**特点**：
- 使用 `@RestController` 注解
- 使用 `@RequestMapping` 定义API路径
- 使用 `@PostMapping`、`@GetMapping` 等定义HTTP方法
- 返回 `Result<T>` 格式的统一响应

**示例**：
```java
@RestController
@RequestMapping("/api/user")
public class UserController {
    @PostMapping("/register")
    public Result<AuthResponse> register(@RequestBody RegisterRequest request) {
        // 调用Service处理业务逻辑
        AuthResponse response = userService.register(request);
        return Result.success(response);
    }
}
```

#### 2. Service 层（业务逻辑层）

**作用**：处理具体的业务逻辑，比如验证数据、调用Repository操作数据库等。

**特点**：
- 使用 `@Service` 注解
- 包含业务逻辑和验证规则
- 调用Repository层操作数据库
- 可以调用其他Service

**示例**：
```java
@Service
public class UserService {
    public AuthResponse register(RegisterRequest request) {
        // 1. 验证用户名是否已存在
        if (userRepository.existsByUsername(request.getUsername())) {
            throw new IllegalArgumentException("用户名已存在");
        }
        // 2. 创建用户
        User user = new User();
        // ... 设置用户属性
        // 3. 保存到数据库
        user = userRepository.save(user);
        // 4. 返回结果
        return AuthResponse.builder()...build();
    }
}
```

#### 3. Repository 层（数据访问层）

**作用**：直接操作数据库，进行增删改查操作。

**特点**：
- 继承 `JpaRepository<Entity, ID>`
- Spring Data JPA 会自动实现基本方法
- 可以自定义查询方法

**示例**：
```java
public interface UserRepository extends JpaRepository<User, Long> {
    // Spring Data JPA 自动提供：save, findById, findAll, delete 等方法
    
    // 自定义查询方法
    Optional<User> findByUsername(String username);
    boolean existsByEmail(String email);
}
```

#### 4. Entity 层（实体层）

**作用**：定义数据库表结构，每个Entity类对应数据库中的一张表。

**特点**：
- 使用 `@Entity` 注解
- 使用 `@Table` 指定表名
- 继承 `BaseEntity` 获得审计字段
- 使用 `@Column` 定义列属性

**示例**：
```java
@Entity
@Table(name = "users")
public class User extends BaseEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    @Column(nullable = false, unique = true)
    private String username;
    
    // ... 其他字段
}
```

## 数据流向

### 完整请求流程

让我们用一个例子来说明：**用户注册**的完整流程

```
1. 前端发送请求
   POST /api/user/register
   {
     "username": "zhangsan",
     "email": "zhangsan@example.com",
     "password": "123456"
   }
   │
   ▼
2. Controller 层接收请求
   UserController.register()
   - 接收 RegisterRequest
   - 调用 UserService.register()
   │
   ▼
3. Service 层处理业务逻辑
   UserService.register()
   - 验证用户名是否已存在（调用 UserRepository.existsByUsername()）
   - 验证邮箱是否已存在（调用 UserRepository.existsByEmail()）
   - 创建User对象，设置属性
   - 保存用户（调用 UserRepository.save()）
   - 生成JWT Token
   - 返回 AuthResponse
   │
   ▼
4. Repository 层操作数据库
   UserRepository.save(user)
   - Spring Data JPA 执行 SQL: INSERT INTO users ...
   │
   ▼
5. 数据库保存数据
   PostgreSQL 执行 INSERT 操作
   │
   ▼
6. 返回结果
   Controller 返回 Result<AuthResponse>
   │
   ▼
7. 前端收到响应
   {
     "code": 200,
     "message": "操作成功",
     "data": {
       "token": "eyJhbGc...",
       "userId": 1,
       "username": "zhangsan"
     }
   }
```

### 关键点说明

1. **Controller 只负责接收和返回**，不处理业务逻辑
2. **Service 处理所有业务逻辑**，包括验证、计算等
3. **Repository 只负责数据库操作**，不关心业务逻辑
4. **Entity 定义数据结构**，对应数据库表

## 模块依赖关系

### 依赖图

```
calotter-start
    ├── 依赖 calotter-common
    ├── 依赖 calotter-user
    ├── 依赖 calotter-inventory
    ├── 依赖 calotter-cooking
    └── 依赖 calotter-health

calotter-user
    └── 依赖 calotter-common

calotter-inventory
    ├── 依赖 calotter-common
    └── 依赖 calotter-user  (因为库存属于家庭，需要User模块的Household)

calotter-cooking
    ├── 依赖 calotter-common
    ├── 依赖 calotter-user  (需要用户和家庭成员信息)
    └── 依赖 calotter-inventory  (需要库存信息来生成推荐)

calotter-health
    ├── 依赖 calotter-common
    ├── 依赖 calotter-user  (需要家庭成员信息)
    ├── 依赖 calotter-inventory  (可以从剩菜记录营养)
    └── 依赖 calotter-cooking  (监听烹饪完成事件)
```

### 依赖说明

1. **所有模块都依赖 calotter-common**
   - 因为都需要使用 Result、BaseEntity 等基础类

2. **calotter-inventory 依赖 calotter-user**
   - 因为库存（食材、厨具等）属于家庭（Household），需要User模块的Household实体

3. **calotter-cooking 依赖 calotter-user 和 calotter-inventory**
   - 需要用户信息来生成个性化推荐
   - 需要库存信息来知道家里有什么食材

4. **calotter-health 依赖其他所有模块**
   - 需要用户信息（家庭成员）
   - 可以从库存模块的剩菜记录营养
   - 监听烹饪模块的完成事件，自动记录营养

5. **calotter-start 依赖所有模块**
   - 作为启动模块，需要加载所有功能

## 快速开始

### 1. 环境要求

- Java 17 或更高版本
- Maven 3.6 或更高版本
- PostgreSQL 数据库

### 2. 配置数据库

编辑 `calotter-start/src/main/resources/application.yml`：

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/calotter
    username: postgres
    password: postgres
```

### 3. 创建数据库

```sql
CREATE DATABASE calotter;
```

### 4. 编译项目

```bash
cd backend-calotter
mvn clean install
```

### 5. 启动应用

```bash
cd calotter-start
mvn spring-boot:run
```

或者直接运行 `CalotterApplication.java` 的 main 方法。

### 6. 验证启动

访问 `http://localhost:8080`，如果看到Spring Boot的默认页面或API响应，说明启动成功。

## 常见问题

### Q1: 为什么要把项目分成这么多模块？

**A**: 模块化设计有以下好处：
- **职责清晰**：每个模块负责一个业务领域
- **便于协作**：不同的人可以同时开发不同模块，减少冲突
- **易于维护**：修改一个模块不会影响其他模块
- **可复用**：通用模块可以被所有业务模块使用

### Q2: 如果我想添加新功能，应该在哪里添加？

**A**: 根据功能类型：
- 如果是新业务功能，在 `calotter-modules` 下创建新模块
- 如果是所有模块共享的功能，添加到 `calotter-common`
- 如果是配置，添加到 `calotter-start` 的 `application.yml`

### Q3: Controller、Service、Repository 的区别是什么？

**A**: 简单理解：
- **Controller**：门卫，接收请求，返回响应
- **Service**：业务员，处理业务逻辑
- **Repository**：仓库管理员，管理数据库

### Q4: 如何查看API接口？

**A**: 启动应用后，可以：
- 使用 Postman 测试API
- 查看各模块的 Controller 类，里面有详细的API说明
- 查看 `docs/postman/` 目录下的 Postman 集合文件
