## 3. 服务器资源优化 (OOM Fix)

### 3.1 问题描述

在尝试后台运行或自动化部署时，进程频发 Status 143 (SIGTERM) 或 137 (SIGKILL)。

**根因：** t2.micro 仅 1GB 内存，Java 应用 + 数据库 + 部署脚本瞬间耗尽内存，触发 Linux OOM Killer

### 3.2 解决方案

配置 2GB Swap (虚拟内存)

```bash
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# 配置 fstab 实现重启自动挂载
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
```

**结果：** `free -h` 显示 Swap 可用，系统稳定性显著提升

## 4. GitHub Actions 自动化脚本攻坚

### 4.1 脚本迭代过程

- **V1 (基础版)**：尝试使用 `appleboy/ssh-action` 远程执行命令，因内存问题失败
- **V2 (暴力清理版)**：加入 `pkill -9` 和 `drop_caches`，导致服务器 IO 卡死，SSH 连接中断
- **V3 (防自杀最终版)**：
  - **问题**：脚本使用 `pgrep -f "java"` 杀进程时，误匹配到了正在执行脚本的 Runner 进程，导致脚本"自杀"
  - **修复**：改用**端口查杀**逻辑

### 4.2 最终脚本逻辑 (deploy.yml)

```yaml
script: |
  # 1. 依赖检查
  sudo apt-get install -y psmisc
  # 2. 精准终止旧进程 (按 8080 端口)
  sudo fuser -k 8080/tcp || true
  # 3. 启动新进程 (注入 Secrets 密码)
  nohup java -jar $JAR_NAME ... --spring.datasource.password=${{ secrets.DB_PASSWORD }} > app.log 2>&1 &
```

## 6. 技术知识点总结 (Lessons Learned)

1. **部署的本质**：不是魔法，只是把本地跑通的命令（`java -jar`）变成自动化脚本在远程跑

2. **低配服生存法则**：t2.micro 等小内存实例，必须配置 Swap，否则 Java 应用必挂

3. **脚本安全性**：在 Shell 脚本中杀进程时，避免使用名称模糊匹配（`pkill name`），按端口查杀 (`fuser -k port`) 永远是最安全的

4. **敏感信息管理**：数据库密码等绝对不能提交到代码库，必须使用 GitHub Secrets 动态注入
