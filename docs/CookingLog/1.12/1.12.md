## 1) 数据库连接故障排查与修复

### 1.1 问题描述

应用启动失败，报错 `FATAL: password authentication failed for user "postgres"`。
初步排查 AWS RDS 控制台显示无活跃实例，但应用日志显示连接拒绝而非超时。

### 1.2 根因分析

* **误判环境**：项目配置原计划连接 RDS，但实际 EC2 实例内部运行了一个本地 PostgreSQL 服务。
* **配置缺失**：手动启动 jar 包时未注入正确的数据库密码，且本地数据库密码未知。

### 1.3 解决方案

* **确认本地服务**：使用 `ps -ef | grep postgres` 确认 EC2 本地运行着 PostgreSQL 16。
* **重置凭据**：
```bash
sudo -u postgres psql
ALTER USER postgres PASSWORD '123456';
CREATE DATABASE calotter;

```


* **验证启动**：使用 CLI 手动带参启动成功，验证了 `--spring.datasource.password`注入方式的可行性。

### 1.4 影响

* ✅ 恢复了后端服务的数据存取能力
* ✅ 明确了生产环境使用 Local DB 的架构（节省成本）

## 2) 服务器资源优化 (OOM Fix)

### 2.1 问题描述

在 GitHub Actions 尝试重启服务时，频发 `Process exited with status 143` (SIGTERM) 或 `137` (SIGKILL)。
这是典型的 **Out Of Memory (OOM)** 现象，t2.micro 仅 1GB 内存，同时运行 Java + PostgreSQL + 部署脚本导致内存溢出。

### 2.2 解决方案

**配置 Swap 虚拟内存**
在 EC2 上划分 2GB 硬盘空间作为虚拟内存，防止内存尖峰导致系统卡死。

```bash
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
# 写入 fstab 实现开机自启

```

**验证**：`free -h` 显示 Swap 空间为 2.0Gi。

### 2.3 影响

* ✅ 彻底解决了部署过程中的随机崩溃问题
* ✅ 增强了服务器运行 Java 应用的稳定性

## 3) CI/CD 脚本重构与防自杀优化

### 3.1 问题描述

GitHub Actions 脚本在执行 `pkill -f "java -jar"` 时，部署任务突然中断，报错连接重置。

**根因分析（Script Suicide）**：

* GitHub Actions 通过 SSH 发送的命令本身也包含 `java -jar` 字符串。
* `pgrep -f` 模糊匹配到了**部署脚本自己的进程**。
* 脚本执行 `kill` 时误杀了自己，导致 SSH 连接意外断开。

### 3.2 解决方案

**重构 `deploy.yml` (Restrat Application Step)**

* **弃用模糊匹配**：不再使用 `pkill` 或 `pgrep`。
* **改用端口查杀**：使用 `fuser -k 8080/tcp` 精准杀掉占用业务端口的进程。
* **环境自动修复**：添加 `apt-get install psmisc` 确保工具存在。
* **动态注入**：通过 GitHub Secrets (`${{ secrets.DB_PASSWORD }}`) 注入密码。

**最终脚本片段：**

```yaml
script: |
  # 安装工具防止报错
  sudo apt-get update && sudo apt-get install -y psmisc
  # 精准杀掉占用 8080 的旧进程
  sudo fuser -k 8080/tcp || true
  # 启动新进程
  nohup java -jar $JAR_NAME ... > app.log 2>&1 &

```

### 3.3 影响

* ✅ 实现了 100% 成功率的自动化部署
* ✅ 解决了脚本"自杀"的隐患
* ✅ 支持无人工干预的后台静默启动

## 6) 技术知识点总结

### 6.1 自动化部署中的"自杀"风险

**问题**：在 SSH 远程脚本中使用 `pkill -f name` 非常危险，因为它可能匹配到正在执行该命令的 Shell 父进程。
**最佳实践**：

* 优先使用 PID 文件管理进程。
* 或者使用 `fuser -k port/tcp` 按端口查杀，这在 Web 应用部署中最为精准安全。

### 6.2 低配服务器 (t2.micro) 优化

**问题**：Java 应用启动瞬间内存消耗大，易触发 Linux OOM Killer。
**最佳实践**：

* 必须配置 Swap (虚拟内存)，建议为物理内存的 1-2 倍。
* 启动脚本中移除 `vm.drop_caches` 等激进的内存清理命令，避免系统 I/O 卡死。

### 6.3 数据库安全与配置

**最佳实践**：

* 生产环境绝对不要在代码 (`application.yml`) 中硬编码密码。
* 使用 CI/CD 平台的 Secrets (如 GitHub Secrets) 在启动命令中动态注入 `--spring.datasource.password`。
