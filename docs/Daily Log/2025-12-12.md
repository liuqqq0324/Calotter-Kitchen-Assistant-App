## Debug: 修复用户注册时的数据库字段不匹配错误

### 问题背景

#### 错误信息
```
Registration failed:
Error updating database. Cause: org.postgresql.util.PSQLException:
ERROR: column "create_dept" of relation "ums_user" does not exist
Position: 72
```

#### 问题原因
1. **实体类继承问题**：`User` 类继承了 `BaseEntity`，`BaseEntity` 包含以下审计字段：
   - `createDept` (创建部门ID)
   - `createBy` (创建者ID)
   - `createTime` (创建时间)
   - `updateBy` (更新者ID)
   - `updateTime` (更新时间)

2. **数据库表结构不匹配**：
   - `calotter/db/ums/ums_user.sql` 中只有 `created_at` 和 `updated_at`（snake_case）
   - `calotter/db/db_init.sql` 中有完整的审计字段，但字段名是 `create_time` 和 `update_time`
   - 实际执行的 SQL 文件（`ums_user.sql`）缺少这些字段

3. **项目需求分析**：
   - 这是一个**个人厨师助手应用**（Personal Sous Chef），不是企业级管理系统
   - `createDept`（创建部门）字段在个人应用中**没有实际意义**
   - `createBy`/`updateBy`（创建者/更新者）字段在个人应用中**意义有限**（用户通常自己操作自己的数据）
   - `createTime`/`updateTime`（创建时间/更新时间）字段**有用**，可以记录用户注册时间和最后修改时间

### 解决方案

**决策：只保留 `create_time` 和 `update_time` 两个时间字段，排除其他审计字段**

#### 修改策略
1. **数据库层面**：修改 SQL 脚本，只包含 `create_time` 和 `update_time`
2. **Java 实体层面**：在 `User` 类中覆盖 `BaseEntity` 的字段，使用 `@TableField(exist = false)` 排除不需要的字段

### 修改的文件清单

#### 1. 数据库脚本文件

##### 文件 1: `calotter/db/ums/ums_user.sql`
**修改内容：**
- 将 `created_at` 改为 `create_time`（与 BaseEntity 的字段名匹配）
- 将 `updated_at` 改为 `update_time`（与 BaseEntity 的字段名匹配）
- 移除其他审计字段（`create_dept`, `create_by`, `update_by`）

**修改前：**
```sql
created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
```

**修改后：**
```sql
create_time TIMESTAMP,
update_time TIMESTAMP,
```

**原因：**
- MyBatis-Plus 会自动将 Java 的 `createTime`（驼峰命名）映射到数据库的 `create_time`（下划线命名）
- 统一使用 `create_time` 和 `update_time` 命名，与 `BaseEntity` 保持一致

##### 文件 2: `calotter/db/db_init.sql`
**修改内容：**
- 在 `ums_user` 表定义中，移除 `create_dept`、`create_by`、`update_by` 字段
- 只保留 `create_time` 和 `update_time` 字段

**修改位置：** 第 720-740 行（`ums_user` 表定义部分）

**修改前：**
```sql
create_dept INT8,
create_by INT8,
create_time TIMESTAMP,
update_by INT8,
update_time TIMESTAMP,
```

**修改后：**
```sql
create_time TIMESTAMP,
update_time TIMESTAMP,
```

#### 2. Java 实体类文件

##### 文件 3: `calotter/calotter-user/src/main/java/com/calotter/user/domain/User.java`
**修改内容：**
- 在 `User` 类中添加字段覆盖方法
- 使用 `@TableField(exist = false)` 注解排除不需要的审计字段
- 只保留 `createTime` 和 `updateTime` 字段的自动填充功能

**添加的代码：**
```java
/**
 * Override BaseEntity fields to exclude audit fields that don't exist in database table.
 * Only create_time and update_time are needed for this personal application.
 */
@TableField(exist = false)
@Override
public Long getCreateDept() {
    return super.getCreateDept();
}

@TableField(exist = false)
@Override
public void setCreateDept(Long createDept) {
    super.setCreateDept(createDept);
}

@TableField(exist = false)
@Override
public Long getCreateBy() {
    return super.getCreateBy();
}

@TableField(exist = false)
@Override
public void setCreateBy(Long createBy) {
    super.setCreateBy(createBy);
}

@TableField(exist = false)
@Override
public Long getUpdateBy() {
    return super.getUpdateBy();
}

@TableField(exist = false)
@Override
public void setUpdateBy(Long updateBy) {
    super.setUpdateBy(updateBy);
}
```

**技术说明：**
- `@TableField(exist = false)` 告诉 MyBatis-Plus 这些字段不存在于数据库表中
- 覆盖 getter/setter 方法确保 Lombok 生成的代码正确
- `createTime` 和 `updateTime` 字段会自动填充（通过 `BaseEntity` 的 `@TableField(fill = FieldFill.INSERT)` 和 `@TableField(fill = FieldFill.INSERT_UPDATE)` 注解）

### 修改后的效果

1. **数据库表结构**：
   - `ums_user` 表只包含 `create_time` 和 `update_time` 两个时间字段

2. **Java 实体类**：
   - `User` 类继承 `BaseEntity`，但排除了不需要的审计字段
   - `createTime` 和 `updateTime` 会自动填充（插入时填充创建时间，更新时填充更新时间）

## 工作内容 3: 修复用户偏好/禁忌/过敏数据持久化问题

### 问题背景

**问题描述：** 
- 用户编辑偏好、禁忌和过敏后，在 profile 和 edit 页面可以看到编辑结果
- 但退出再登录后，数据就丢失了
- 数据无法正确持久化到数据库

### 问题分析

#### 1. 字段名不匹配问题（已修复）

**问题：**
- `IntakeRecordMapper.xml` 中使用了 `created_time` 字段
- 但数据库实际字段名是 `create_time`
- 导致查询时报错：`ERROR: column "created_time" does not exist`

**修复：**
- 修复了 `calotter/calotter-homepage/src/main/resources/mapper/homepage/IntakeRecordMapper.xml`
- 将 3 处 `ORDER BY created_time DESC` 改为 `ORDER BY create_time DESC`

#### 2. 空值处理问题（核心问题）

**问题：**
- `updatePreferences` 方法中，当 `dietaryType`、`spiceLevel` 或 `cookingTimePreference` 为空字符串时
- 仍会创建无效的 preference 记录（如 `"dietaryType:"`、`"spiceLevel:"` 等）
- `updateTaboos` 和 `updateAllergies` 方法中，空字符串也会创建无效的 restriction 记录

**影响：**
- 数据库中产生大量无效记录
- 可能导致数据查询和显示异常
- 影响数据持久化的正确性

### 解决方案

#### 修改文件
- `calotter/calotter-user/src/main/java/com/calotter/user/api/UmsApiController.java`

#### 具体修改

**1. `updatePreferences` 方法（285-308行）**

**修改前：**
