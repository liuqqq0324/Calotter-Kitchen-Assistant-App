# COMPSCI 778 – Journal Two

## 项目描述 / Project Description

Personal Sous Chef（个人厨师助手）是一个全栈智能移动应用项目，旨在通过人工智能技术帮助用户更高效地管理厨房食材、生成个性化食谱，并提供实时烹饪指导。该项目采用前后端分离的架构设计，前端使用 Flutter 框架开发跨平台移动应用，后端采用 Spring Boot 多模块架构，并集成 AI 引擎实现食材识别和智能食谱生成功能。

项目的核心价值在于解决现代家庭厨房管理中的痛点：食材浪费、食谱选择困难、烹饪过程缺乏指导等问题。通过机器学习图像识别技术自动识别食材（由团队成员负责开发），结合用户的饮食偏好、过敏信息和库存情况，系统能够智能推荐最适合的食谱，并在烹饪过程中提供分步骤的实时指导。

## 项目背景与研究 / Project Background and Research

### 研究背景

在当今快节奏的生活中，家庭厨房管理面临着诸多挑战。根据相关研究，家庭食物浪费是一个全球性问题，而缺乏有效的食材管理和食谱规划是导致浪费的主要原因之一。传统的食材管理方式依赖人工记录，效率低下且容易出错。同时，用户在选择食谱时往往面临信息过载的问题，难以找到既符合个人偏好又能够充分利用现有食材的食谱。

### 技术研究现状

在食材识别领域，计算机视觉技术已经取得了显著进展。深度学习模型，特别是 YOLO（You Only Look Once）系列模型，在目标检测任务中表现出色，能够实现实时、高精度的物体识别。本项目团队正在使用 YOLO 模型进行食材识别，通过训练自定义数据集来识别常见的厨房食材。这部分工作由团队成员负责，我主要负责后端库存管理和用户管理模块的开发。

在食谱生成方面，生成式 AI 技术（如大型语言模型）已经能够根据给定的食材和用户偏好生成高质量的食谱内容。通过构建合适的提示词（prompt）和上下文信息，AI 可以生成个性化的、详细的烹饪步骤。

在系统架构方面，RESTful API 设计已经成为现代 Web 应用的标准。REST（Representational State Transfer）架构风格提供了统一接口、无状态操作、可缓存响应等特性，使得系统具有良好的可扩展性和可维护性（Fielding, 2000）。本项目采用 RESTful API 设计，确保前后端之间的数据交互清晰、规范。

### 相关技术文献

- **REST 架构风格**：Fielding (2000) 在其博士论文中提出了 REST 架构的六个约束条件，包括统一接口、无状态操作、可缓存响应等，这些原则指导了本项目的 API 设计。

- **深度学习在图像识别中的应用**：YOLO 系列模型在实时目标检测任务中表现出色，适合移动端和边缘设备的部署需求。团队正在基于此技术开发食材识别功能。

- **微服务架构**：本项目采用 Spring Boot 多模块架构，借鉴了微服务的设计思想，将不同业务功能模块化，提高了系统的可维护性和可扩展性。

## 项目目标与目标 / Project Goals and Objectives

### 核心目标

1. **开发一个功能完整的智能厨房管理应用**
   - 实现用户账户管理和个性化设置
   - 提供直观、易用的用户界面
   - 支持跨平台部署（iOS 和 Android）

2. **实现基于 AI 的食材识别功能**
   - 通过图像/视频扫描自动识别食材（由团队成员负责，使用 YOLO 模型）
   - 支持用户手动修正识别结果
   - 提供高精度的食材识别准确率

3. **提供智能食谱生成和推荐**
   - 基于用户库存自动生成可用食谱

   - 提供食谱搜索和筛选功能

4. **实现烹饪过程指导功能**
   - 提供分步骤的烹饪指南
   - 支持免提模式（语音导航）
   - 内置计时器功能

**API 接口实现**

- ✅ **用户信息管理**：实现用户信息获取和更新 API（`GET /api/user`、`PUT /api/user`）
- ✅ **个性化设置**：实现用户偏好、禁忌、过敏信息的获取和更新 API
  - `GET/PUT /api/user/preferences` - 用户偏好管理
  - `GET/PUT /api/user/taboos` - 用户禁忌管理
  - `GET/PUT /api/user/allergies` - 用户过敏管理
- ✅ **家庭管理**：实现家庭创建、查询、邀请码管理等功能
- ✅ **家庭成员管理**：实现家庭成员的添加、查询、更新等功能

**数据存储设计**
- ✅ 使用 PostgreSQL 数据库进行数据持久化
- ✅ 使用 JPA 进行数据访问，采用 Spring Data JPA 简化开发
- ✅ 使用 JSONB 字段存储用户设置和偏好信息，提高灵活性
- ✅ 实现数据验证（Jakarta Validation）确保数据完整性

**与其他模块的配合**
- ✅ **与库存模块配合**：库存模块依赖用户模块的 `Household` 实体，所有库存数据都关联到家庭
- ✅ **与烹饪模块配合**：烹饪模块使用用户模块的 `User`、`FamilyMember`、`HealthGoal` 实体来构建个性化推荐上下文
- ✅ **与健康模块配合**：健康模块使用 `FamilyMember` 实体来记录家庭成员的营养摄入

**应用交互与商业化思考**

在设计用户模块时，我特别考虑了应用交互和商业化需求：

1. **注册机制设计**：我设计了"注册才能被邀请到家庭中"的机制。这意味着用户必须先注册账户，才能通过邀请码加入家庭。这个设计有以下优势：
   - **用户增长**：确保所有使用家庭功能的用户都是注册用户，有助于建立用户账户体系
   - **数据完整性**：注册用户有完整的用户档案，便于提供个性化服务
   - **商业化基础**：注册用户体系为未来的付费功能、会员体系等商业化策略奠定基础
   - **安全性**：注册机制可以配合邮箱验证，提高账户安全性

2. **家庭管理设计**：通过邀请码机制，家庭管理员可以邀请其他用户加入家庭。被邀请的用户必须是已注册用户，这确保了：
   - 家庭成员都有完整的用户档案
   - 可以追踪每个成员的使用情况
   - 便于实现家庭级别的功能（如家庭库存共享、家庭健康报告等）

3. **数据隔离与共享**：库存数据属于家庭（Household），但每个家庭成员都可以访问。这种设计既保证了数据的共享性，又通过家庭管理实现了数据隔离，为多用户场景提供了良好的支持。

#### 2. 库存模块（calotter-inventory）开发

**实体设计与数据模型**
- ✅ **Ingredient（食材实体）**：实现食材库存管理，包括数量、单位、过期日期、存放位置等
- ✅ **HouseholdSpice（调料实体）**：实现调料管理
- ✅ **HouseholdUtensil（厨具实体）**：实现厨具管理
- ✅ **LeftoverDish（剩菜实体）**：实现剩菜管理，为健康模块提供数据支持
- ✅ **标准库实体**：实现 `StandardIngredient`、`StandardSpice`、`StandardUtensil`、`RefAllergen` 等标准库，确保数据一致性

**API 接口实现**
- ✅ **食材管理**：
  - `POST /api/inventory/ingredients` - 创建食材
  - `PUT /api/inventory/ingredients/{id}` - 更新食材
  - `GET /api/inventory/ingredients` - 查询食材列表
  - `DELETE /api/inventory/ingredients/{id}` - 删除食材
  - `POST /api/inventory/ingredients/{id}/deduct` - 扣减食材库存（烹饪后使用）
- ✅ **调料管理**：实现调料的增删改查 API
- ✅ **厨具管理**：实现厨具的增删改查 API
- ✅ **剩菜管理**：实现剩菜的增删改查 API
- ✅ **标准库查询**：实现标准食材、调料、厨具的搜索 API

**核心功能实现**
- ✅ **库存扣减功能**：实现烹饪后自动扣减食材数量的功能，确保库存数据准确性
- ✅ **标准库关联**：所有库存数据都关联到标准库，确保数据一致性和可查询性
- ✅ **家庭隔离**：所有库存数据都关联到家庭（Household），实现数据隔离和共享

**与其他模块的配合**

库存模块是整个系统的核心数据源之一，与多个模块紧密配合：

1. **与用户模块配合**：
   - 库存数据关联到 `Household` 实体，确保只有家庭成员可以访问家庭库存
   - 通过家庭管理实现多用户场景下的数据共享和隔离

2. **与烹饪模块配合**：
   - 烹饪模块使用库存模块的 `Ingredient`、`HouseholdSpice`、`HouseholdUtensil` 实体来构建"厨房快照"
   - 厨房快照包含家庭当前可用的所有食材、调料和厨具，用于生成 AI 烹饪推荐
   - 烹饪完成后，通过库存扣减 API 自动减少食材数量
   - 剩菜数据由烹饪模块创建，存储在库存模块的 `LeftoverDish` 实体中

3. **与健康模块配合**：
   - 健康模块使用库存模块的 `LeftoverDish` 实体来记录营养摄入
   - 当用户食用剩菜时，健康模块可以从剩菜的营养信息中创建营养日志
   - 这种设计实现了数据的复用，避免了数据冗余

**应用交互思考**

在设计库存模块时，我考虑了以下交互场景：

1. **图像识别集成**：虽然图像识别功能由团队成员负责开发，但我在设计库存 API 时考虑了接收识别结果的场景。当 AI 引擎识别出食材后，可以通过库存 API 创建食材记录，用户可以在创建前进行确认和修正。

2. **库存预警**：设计了过期日期字段，未来可以实现过期预警功能，提醒用户及时使用即将过期的食材，减少浪费。

3. **库存扣减自动化**：实现了库存扣减功能，当用户完成烹饪后，系统可以自动减少使用的食材数量，保持库存数据的准确性。

#### 模块依赖关系设计

我参与设计的模块依赖关系确保了系统的可维护性和可扩展性：

```
calotter-inventory（库存模块）
    ├── 依赖 calotter-common（通用功能）
    └── 依赖 calotter-user（需要 Household 实体）

calotter-user（用户模块）
    └── 依赖 calotter-common（通用功能）

calotter-cooking（烹饪模块）
    ├── 依赖 calotter-common
    ├── 依赖 calotter-user（需要用户和家庭成员信息）
    └── 依赖 calotter-inventory（需要库存信息）

calotter-health（健康模块）
    ├── 依赖 calotter-common
    ├── 依赖 calotter-user（需要家庭成员信息）
    ├── 依赖 calotter-inventory（可以从剩菜记录营养）
    └── 依赖 calotter-cooking（监听烹饪完成事件）
```

这种依赖关系设计确保了：
- **模块解耦**：每个模块职责清晰，相互独立
- **数据共享**：通过实体引用实现数据共享，避免数据冗余
- **事件驱动**：健康模块通过事件监听机制与烹饪模块交互，避免直接依赖

### 技术挑战与解决方案

**预期挑战**

1. **模块间数据一致性**
   - **挑战**：多个模块共享数据（如 Household、Ingredient），需要确保数据一致性
   - **解决方案**：通过实体引用而非数据复制，使用事务确保数据一致性，设计清晰的数据所有权

2. **JSONB 字段查询性能**
   - **挑战**：用户偏好、设置等数据存储在 JSONB 字段中，复杂查询可能性能不佳
   - **解决方案**：为常用查询添加 GIN 索引，考虑将热点数据提取到独立字段，使用 PostgreSQL 的 JSONB 查询优化

3. **家庭管理的权限控制**
   - **挑战**：需要实现细粒度的权限控制（家庭管理员、普通成员等）
   - **解决方案**：在 FamilyMember 实体中添加角色字段，在 API 层实现权限检查，使用 Spring Security 进行权限管理

4. **库存数据的实时同步**
   - **挑战**：多个用户可能同时修改库存数据，需要处理并发冲突
   - **解决方案**：使用乐观锁机制，实现库存变更的版本控制，添加库存变更日志

## 总结 / Summary

Personal Sous Chef 项目目前已经完成了后端核心架构的搭建，我主要负责的用户模块和库存模块的主要功能已经实现。用户模块实现了完整的用户账户管理、个性化设置、家庭管理等功能，并设计了"注册才能被邀请到家庭中"的机制，为商业化策略奠定了基础。库存模块实现了食材、调料、厨具、剩菜的完整管理功能，并通过与用户模块、烹饪模块、健康模块的紧密配合，为整个系统提供了核心数据支持。

在开发过程中，我特别注重模块间的配合和系统架构的设计。通过清晰的模块依赖关系和事件驱动机制，确保了系统的可维护性和可扩展性。同时，我也考虑了应用交互和商业化需求，设计了注册机制、家庭管理等功能，为未来的用户增长和商业化策略提供了基础。

项目的下一步重点是完善模块间的集成，优化 API 接口性能，并实现前端与后端的完整集成。同时，团队正在进行的 AI 图像识别功能（使用 YOLO 模型）和健康管理模块的开发也在稳步推进。通过团队协作，我们希望能够为用户提供一个真正有用的智能厨房管理工具。

这个项目不仅是一个技术实践，更是对如何利用现代技术解决实际生活问题的探索。通过结合后端系统设计、数据库设计、模块化架构等多个领域，我不仅提升了技术能力，也加深了对系统设计和商业化的理解。
