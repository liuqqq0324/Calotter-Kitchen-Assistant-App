## 临时方案实现（用于前后端连通和数据库交互测试）

### 问题
为了能够立即测试前后端连通和数据库交互，需要实现一个临时的token验证机制。否则所有需要认证的API都会返回"Unauthorized"错误。

### 实现方案
实现了基于内存缓存的临时token验证机制，让系统能够立即工作。

#### 更新的文件

1. **TokenUtils.java** - 添加了临时缓存机制
   - 添加了 `TokenCache` 内部类：使用 `ConcurrentHashMap` 存储 token -> userId 映射
   - 添加了 `cacheToken()` 方法：存储token到userId的映射
   - 添加了 `removeToken()` 方法：移除token（用于登出）
   - 更新了 `extractUserIdFromToken()` 方法：从缓存中查找userId

2. **UmsApiController.java** - 在登录和登出时管理token缓存
   - 登录时：调用 `TokenUtils.cacheToken()` 缓存token和userId的映射
   - 登出时：调用 `TokenUtils.removeToken()` 移除token

#### 技术细节

**实现方式：**
- 使用 `ConcurrentHashMap` 作为内存缓存
- 登录成功后自动缓存 token -> userId 映射
- 登出时自动移除token
- Token验证时从缓存中查找userId

**优点：**
- ✅ 简单快速，可以立即使用
- ✅ 前后端可以立即连通
- ✅ 数据库交互可以正常测试
- ✅ 不需要额外的依赖（Redis等）

**缺点：**
- ⚠️ 服务重启后数据会丢失（需要重新登录）
- ⚠️ 多实例部署时无法共享（每个实例有独立的缓存）
- ⚠️ 不适合生产环境（应该使用Redis或数据库）

#### 当前系统状态

**前后端连通：** ✅ 可以
- 登录后token会被缓存
- 前端调用API时，后端能从缓存中获取userId
- API可以正常响应

**数据库交互：** ✅ 可以
- 后端已配置数据库连接
- Mapper已实现
- 可以正常读写数据库

#### 使用流程

1. **用户登录**
   ```
   POST /api/ums/auth/login
   → 返回token和userId

   ```

2. **调用需要认证的API**

3. **用户登出**

   ```
   Header: Authorization: Bearer <token>
   → 从缓存中移除token
   ```

# 为什么不用 Feign，而是直接查询数据库？

## 原因分析

### 同一数据库连接
- **calotter-homepage**：连接 `sous_chef` 数据库，默认 schema → `sous_chef_hp`
- **calotter-user**：连接 `sous_chef` 数据库，默认 schema → `sous_chef_ums`
- 两个服务连接的是 **同一个数据库**，只是默认 schema 不同。

### 可以直接跨 schema 查询
- SQL 中可直接访问：`sous_chef_ums.ums_user`
- MyBatis Plus 支持通过 `@TableName(schema = "...")` 指定 schema

---

## 优势（直接查数据库 vs Feign）

| 方案 | Feign 调用 | 直接数据库查询 |
|------|-------------|----------------|
| 复杂度 | 需要 Feign 配置 | 更简单，只需写 Mapper |
| 性能 | 有网络延迟 | 无网络延迟，更快 |
| 可靠性 | 依赖服务间通信 | 不依赖其他服务，更稳 |
| 配置 | 需要服务端点地址 | 不需要额外配置 |
| 可用性 | 服务挂掉会影响 | 不受其他服务状态影响 |

➡ **结论：直接查数据库更符合当前架构，简单、高效、可靠。**

# Homepage模块userId获取方式修改（2025-12-11）

## 修改原因

为了与其他模块（如inventory模块）保持一致，将homepage模块的userId获取方式从**Authorization Header（Bearer Token）**改为**URL查询参数（@RequestParam）**。

## 修改内容

### 1. 后端代码修改

#### NutritionController.java
- ✅ 移除 `TokenUtils` 导入和使用
- ✅ 将 `@RequestHeader(value = "Authorization")` 改为 `@RequestParam("userId") Long userId`
- ✅ 移除token验证相关的null检查和错误处理
- ✅ 更新API路径注释，添加userId参数说明

**修改前：**
```java
@GetMapping("/targets/weekly")
public R<...> getWeeklyNutritionTargets(
        @RequestHeader(value = "Authorization", required = false) String authHeader) {
    Long userId = TokenUtils.extractUserIdFromToken(authHeader);
    if (userId == null) {
        return R.fail("Unauthorized: Invalid or missing token");
    }
    // ...
}
```

**修改后：**
```java
@GetMapping("/targets/weekly")
public R<...> getWeeklyNutritionTargets(
        @RequestParam("userId") Long userId) {
    // ...
}
```

#### IntakeController.java
- ✅ 移除 `TokenUtils` 导入和使用
- ✅ 所有方法（`getTodayIntakes`, `updateIntakePercentage`, `addManualIntake`）都改为使用 `@RequestParam("userId")`
- ✅ 移除token验证相关的错误处理
- ✅ 更新API路径注释

**修改的方法：**
- `getTodayIntakes`: `GET /api/intake/today?source={source}&userId={userId}`
- `updateIntakePercentage`: `PATCH /api/intake/{intake_id}?userId={userId}`
- `addManualIntake`: `POST /api/intake/manual?userId={userId}`

### 2. API文档修改

#### homepage-api.md
- ✅ 所有API的请求格式从 `Authorization: Bearer <accessToken>` 改为 `userId: <user_id>` 查询参数
- ✅ 更新了5个API的文档：
  1. Get Weekly Nutrition Targets: `GET /api/nutrition/targets/weekly?userId={userId}`
  2. Get Weekly Nutrition Summary: `GET /api/nutrition/summary?period=week&userId={userId}`
  3. Get Today Intakes: `GET /api/intake/today?source={source}&userId={userId}`
  4. Update Intake Percentage: `PATCH /api/intake/{intake_id}?userId={userId}`
  5. Add Manual Intake: `POST /api/intake/manual?userId={userId}`

### 3. Postman Collection修改

#### Calotter-Homepage-API-v2.postman_collection.json
- ✅ 移除所有API请求中的 `Authorization` header
- ✅ 为所有API添加 `userId` 查询参数
- ✅ 更新登录接口的测试脚本：从保存 `access_token` 改为保存 `user_id`
- ✅ 更新collection变量：移除 `access_token`，添加 `user_id`
- ✅ 更新所有API的URL格式，添加 `?userId={{user_id}}`

**修改的API：**
- 获取周营养目标
- 获取周营养摘要
- 获取今日摄入（全部/菜谱来源/手动来源）
- 更新摄入百分比
- 添加手动摄入
