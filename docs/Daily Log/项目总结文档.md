# 项目总结文档

## 📋 项目概述

本项目是一个基于 Spring Boot 的智能烹饪助手后端系统，采用模块化设计，主要包含用户管理、库存管理和烹饪推荐三大核心功能模块。

---

## 📦 已实现的实体类（Entity）

- **功能**: 所有实体类的基类
- **字段**:

  - `updateBy` - 更新人ID（Long类型）
- **特性**: 使用 JPA Auditing 自动管理审计字段

- **表名**: `ref_standard_ingredients`

## 🔧 已实现的业务逻辑

### 1. 用户认证服务（UserService）

#### 功能概述
提供用户注册和登录功能，包含密码加密和 JWT Token 生成。

#### 核心方法

**register(RegisterRequest request)**
- **功能**: 用户注册
- **流程**:
  1. 检查用户名唯一性
  2. 检查邮箱唯一性
  3. 使用 `PasswordEncoder` 加密密码
  4. 创建用户实体（默认角色ROLE_USER，状态可用，未完成引导）
  5. 保存用户
  6. 生成 JWT Token
  7. 返回认证响应（包含Token和用户信息）
- **异常处理**: 用户名或邮箱已存在时抛出 `IllegalArgumentException`

**login(LoginRequest request)**
- **功能**: 用户登录
- **流程**:
  1. 支持用户名或邮箱登录（优先用户名，失败则尝试邮箱）
  2. 验证密码（使用 `PasswordEncoder.matches`）
  3. 检查用户状态（未激活或封禁时拒绝登录）
  4. 生成 JWT Token
  5. 返回认证响应
- **异常处理**: 用户名/密码错误或账户状态异常时抛出 `IllegalArgumentException`

### 2. JWT Token 服务（JwtService）

#### 功能概述
提供 JWT Token 的生成、解析和验证功能。

#### 核心方法

**generateToken(Long userId, String username)**
- 生成包含 userId 和 username 的 JWT Token
- 配置了 issuer、audience 和过期时间

**extractUsername(String token)**
- 从 Token 中提取用户名

**extractUserId(String token)**
- 从 Token 中提取用户ID

**validateToken(String token)**
- 验证 Token 是否有效且未过期

### 3. 烹饪上下文构建服务（CookingContextBuilderService）

#### 功能概述
构建用于 AI 烹饪推荐的完整上下文，整合家庭成员信息、健康目标、库存状态等多维度数据。

#### 核心方法

**buildContext(CookingGenerationRequest req)**
- **功能**: 构建 AI 请求上下文的主入口
- **流程**:
  1. 根据成员ID列表获取家庭成员数据
  2. 调用 `processDinersAndGoals` 处理用餐者画像与营养目标
  3. 调用 `processInventory` 处理厨房库存
  4. 构建任务设置（菜品数量、最大时间、难度、目标菜系）
  5. 组装并返回 `AiCookingContext` 对象

**processDinersAndGoals(...)**
- **功能**: 处理所有用餐者的画像与营养目标
- **核心逻辑**:
  1. **处理家庭成员**:
     - 收集硬性限制（过敏原、饮食风格）加入全局避让列表
     - 统计软性偏好（忌口、口味、菜系）频率
     - 提取个人忌口用于个人画像
     - 根据健康目标计算营养需求（本餐按35%比例分配，无目标则使用默认值）
     - 构建成员花名册（DinerSlot）
  2. **处理客人**:
     - 收集过敏原和偏好
     - 使用默认营养值（700卡路里）
     - 加入花名册
  3. **冲突分析**:
     - 分析忌口冲突（所有人都不吃→全局屏蔽，部分人不吃→记录冲突）
     - 分析菜系偏好（半数以上喜欢→共同喜好）
  4. **营养累加**:
     - 汇总所有用餐者的营养目标（卡路里范围、蛋白质、脂肪、碳水、纤维）
  5. **返回**: `DinerProfile` 和 `CompositeNutritionalGoal`

**processInventory(Long householdId)**
- **功能**: 处理厨房库存快照
- **核心逻辑**:
  1. **处理食材**:
     - 查询数量大于0的食材
     - 计算剩余保质期（距离过期日期的天数）
     - 分类处理：
       - 3天内过期 → 标记为 URGENT，加入优先级列表
       - 其他 → 标记为 NORMAL，加入普通列表
       - 无过期日期 → 标记为 NORMAL
  2. **处理调料**:
     - 查询可用（isAvailable=true）的调料
     - 提取调料名称列表
  3. **处理厨具**:
     - 查询可用（isAvailable=true）的厨具
     - 提取厨具名称列表
  4. **返回**: `KitchenSnapshot`（包含优先级食材、普通食材、可用调料、可用厨具）

## 🌐 API 端点

### 用户模块（UserController）

#### POST /api/user/register
- **功能**: 用户注册
- **请求体**: `RegisterRequest` (username, email, password)
- **响应**: `Result<AuthResponse>`
- **返回内容**: Token, userId, username, email, role

#### POST /api/user/login
- **功能**: 用户登录
- **请求体**: `LoginRequest` (usernameOrEmail, password)
- **响应**: `Result<AuthResponse>`
- **返回内容**: Token, userId, username, email, role

### 烹饪模块（CookingController）

#### POST /api/cooking/generate-context
- **功能**: 生成烹饪上下文
- **请求体**: `CookingGenerationRequest`
  - `memberIds` - 家庭成员ID列表
  - `guests` - 客人信息列表（可选）
  - `dishCount` - 菜品数量
  - `maxTimeMinutes` - 最大时间（分钟）
  - `difficulty` - 难度
  - `targetCuisines` - 目标菜系列表（可选）
- **响应**: `Result<AiCookingContext>`
- **返回内容**: 完整的AI烹饪上下文（包含食客画像、营养目标、厨房快照、任务设置）

## 📝 核心业务场景

### 场景1: 用户注册和登录
1. 用户通过 `/api/user/register` 注册账户
2. 系统验证用户名和邮箱唯一性
3. 密码加密后保存
4. 生成并返回 JWT Token
5. 用户使用 Token 进行后续请求认证

### 场景2: 智能烹饪推荐上下文构建
1. 用户发起烹饪请求（指定用餐成员、客人、偏好等）
2. 系统调用 `CookingContextBuilderService.buildContext` 构建上下文
3. 系统收集所有用餐者的：
   - 过敏原和饮食限制（硬性限制）
   - 口味和菜系偏好（软性偏好）
   - 健康目标和营养需求
4. 系统分析偏好冲突（全局避让 vs 部分冲突）
5. 系统查询厨房库存，分类处理临期食材
6. 系统汇总营养目标（累加所有用餐者需求）
7. 返回完整的 AI 烹饪上下文，用于后续 AI 推荐

## 🎯 项目亮点

1. **智能冲突分析**: 自动分析多用餐者的偏好冲突，区分硬性限制和软性偏好
2. **营养目标计算**: 基于个人健康目标智能计算复合营养需求
3. **库存优先级管理**: 自动识别临期食材（3天内），优先推荐使用
4. **灵活的偏好系统**: 使用 JSONB 存储灵活的偏好数据，支持扩展
5. **模块化架构**: 清晰的模块划分，便于维护和扩展
6. **完整的审计机制**: JPA Auditing 自动记录创建和修改信息
