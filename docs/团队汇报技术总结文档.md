## 第三部分：理论与实践的结合 (5分钟)

### 3.1 开发方法论：Agile/Scrum 与迭代逻辑

#### 迭代规划与节奏

项目采用 **Agile/Scrum** 思路，按 Sprint 划分阶段，每个 Sprint 有明确目标与可交付成果。

| 阶段 | 目标 | 主要产出 | 对应代码/文档依据 |
|------|------|----------|-------------------|
| **Sprint 1（MVP 最小可行性产品）** | 验证核心流程、打通前后端与数据流 | 用户注册/登录、库存 CRUD、基础食谱展示、AI 生成菜单入口 | `docs/Sprint 1/`、`docs/开发进度 - Development Progress.md`、后端多模块 + Flutter 页面框架 |
| **Sprint 2（UI 渲染与功能优化）** | 提升体验、完善功能、稳定联调 | 手绘风格 UI、语音/手势控制、收藏与烹饪流程、健康目标、自动化部署 | `docs/项目总结 - Project Summary.md`、`frontend-app/项目服务总结.md`、`.github/workflows/deploy.yml` |

#### 迭代逻辑在代码中的体现

- **Sprint 1**  
  - **后端**：Maven 多模块（calotter-common / calotter-modules / calotter-start），user、inventory、cooking、health 四大业务模块先行实现核心 API。  
  - **前端**：Flutter 页面骨架（登录/注册、首页、食谱、扫描、库存、个人资料、设置），先以假数据跑通流程。  
  - **数据**：统一数据交换格式文档（`Personal Chef Data Exchange JSON Format UMS.md`）约束前后端与 AI 的 JSON 契约。

- **Sprint 2**  
  - **前端**：`lib/features/` 下按功能拆分（profile、inventory、recipes 等），引入 Otter 主题、Sketchy 按钮、专注模式、语音/手势等。  
  - **后端**：API 路径统一（如 `/api/ai/generate-menus`、`/api/recipes/favorite`）、`Result<T>` 统一响应、householdId 贯穿烹饪与收藏。  
  - **部署**：GitHub Actions 实现 Push 即构建、上传 Jar 到 AWS、重启服务，形成可重复的发布节奏。

### 3.2 技术决策背后的理论

#### 为什么选择 PostgreSQL？

- **关系型建模与关联查询**  
  业务强依赖「用户 → 家庭 → 家庭成员」「家庭 → 库存（食材/调料/厨具）」「库存 → 标准库」「烹饪会话 → 菜品 → 营养」等多层关系。PostgreSQL 提供：
  - 外键与事务，保证数据一致性和库存扣减等操作的原子性。
  - 多表 JOIN、聚合、分页，满足复杂查询（如按家庭查库存、按成员查健康目标）。

- **项目中的具体体现**  
  - 实体关系：`User` ↔ `Household` ↔ `FamilyMember`；`Household` → `Ingredient` / `HouseholdSpice` / `HouseholdUtensil`；`Ingredient` 关联 `StandardIngredient` 等（见 `docs/backend-architecture-docs/`、`docs/Daily Log/项目总结文档.md`）。  
  - 使用 **JSONB** 存储灵活结构（如用户 `settings`、家庭成员 `preferences`、`dietaryStyles`），在关系型基础上兼顾扩展性。

- **一句话**：选 PostgreSQL 是为了在**食材—家庭—用户—食谱—营养**的强关联场景下，用关系型模型保证一致性与可维护性。

#### 为什么使用 AWS EC2 部署而不是本地部署？

- **可扩展性 (Scalability)**  
  - 应用以单机 Jar 部署于 EC2，当前阶段通过垂直扩展（实例规格、JVM 参数）即可应对负载。  
  - 数据库在 EC2 本机（同一 VPC），减少延迟；后续若流量增长，可平滑迁移到 RDS 或增加实例，无需改应用架构。

- **可用性 (Availability)**  
  - 服务 24 小时在线，前端通过固定公网 IP（如 `16.176.170.214:8080`）访问，不依赖任何人电脑开机。  
  - 文档（`docs/awsLog/1.12-自动化部署说明手册.md`）明确：前端可直接连云端后端，无需本地起 Java/数据库，便于多人联调与演示。

- **与 CI/CD 的结合**  
  - `.github/workflows/deploy.yml`：Push 触发 → Maven 构建 → SCP 上传 Jar 至 `/opt/calotter` → SSH 执行脚本释放 8080、注入环境变量、`nohup java -jar` 启动。  
  - 敏感信息（如 `DB_PASSWORD`、`GEMINI_API_KEY`、`GROQ_API_KEY`）通过 GitHub Secrets 注入，避免写进代码，符合生产环境安全实践。

- **一句话**：选 EC2 是为了**可扩展性、高可用性以及与 CI/CD 的衔接**，让团队在「云端统一环境」下协作与交付。

## 第四部分：职业素养与实习反思 (13分钟)

### 4.1 解决问题的能力 (Problem Solving)

#### 前后端 JSON 字段对齐

- **问题**：前端期望的字段名、嵌套结构与后端 DTO/API 返回不一致，导致解析失败或展示错误。  
- **应对**：  
  - 用**统一数据交换文档**（`Personal Chef Data Exchange JSON Format UMS.md`）约定 F↔B、B↔AI 的 JSON 格式（如 inventory 的 `amount_value`/`amount_unit`、菜谱的 `ingredients` 结构）。  
  - 前后端共用「同一份契约」开发；后端使用 DTO（如 `IngredientRequest`/`IngredientResponse`、`MenuDTO`、`AiRecipeResponse`）序列化，前端按文档字段解析。  
  - 联调时出现路径或字段不一致（如 `docs/CookingLog/12.16/cooking-modifications-summary.md` 中 API 路径统一为 `/api/ai/generate-menus`、`Result<T>` 包装），通过**修改一端并同步文档**解决，避免口头约定。

#### AWS Bedrock 限制与方案调整

- **背景**：曾集成 AWS Bedrock 作为 AI 菜单生成选项（见 `docs/CookingLog/1.7/cooking-modifications-1.7.md`），需 Bearer token、区域与模型配置。  
- **限制与决策**：  
  - 受限于 Bedrock 的可用区、配额或运维成本，团队将**默认生产方案**改为 **Spring AI + Gemini**（见 `calotter-cooking/.../config/AiConfig.java` 注释：「已废除 Mock、Groq 和 Bedrock」）。  
  - 营养估算仍保留 **Groq** 作为可选 provider（`application.yml` 中 `nutrition.provider`、`groq` 配置），与 Gemini 形成互补。  
- **收获**：在**约束条件下做技术选型**（可用性、成本、团队熟悉度），并保留配置化切换能力（如 `ai.api.mode`），便于未来再评估 Bedrock。

#### Code Refactoring 与代码质量

- **后端**：  
  - **模块化**：common（Result、BaseEntity、全局异常）与 user、inventory、cooking、health 分离，降低耦合。  
  - **分层**：Controller → Service → Repository → Entity，职责清晰；DTO 与 Entity 分离，避免 API 与表结构强绑定。  
  - **重构记录**：如 `docs/Daily Log/重构方案.md` 中从「单块+MyBatis」到「JPA + 多模块」的迁移；`user-log` 下有多篇重构与修复日志（如偏好标准库统一、健康目标同步修复）。  
- **前端**：  
  - 按 feature 组织（`lib/features/`），共享组件与主题（`lib/shared/`、`lib/theme/`），服务层（`lib/services/`）集中封装 API 与 `Result<T>` 解析。  
- **效果**：可维护性提升，多人可并行改不同模块；接口契约清晰，便于测试与联调。

### 4.2 团队协作与属性

#### Git 版本管理

- **分支与部署**：  
  - 当前部署由 `.github/workflows/deploy.yml` 在指定分支（如 `yhua`）的 Push 上触发，实现「提交即部署」。  
  - 文档提醒：多人同时 Push 时后提交会覆盖先提交的版本，建议在主干或发布分支上做重要测试并同步（见 `docs/awsLog/1.12-自动化部署说明手册.md`）。  

- **协作习惯**：  
  - 通过 feature/模块分支开发，再合并到集成分支；敏感信息仅通过 GitHub Secrets 传递，不提交密钥与密码。

#### 技术文档同步 5 人开发

- **文档结构**（`docs/`）：  
  - **架构与模块**：`backend-architecture-docs/`（后端架构总览、user/inventory/cooking/health 各模块文档）— 新人或接手某模块时快速理解。  
  - **API 与数据**：`Personal Chef Data Exchange JSON Format UMS.md`、`recipe_api.md`、`inventory-api-postman.md`、`postman/` 下各 Postman 集合 — 前后端与测试对齐接口与 JSON。  
  - **迭代与变更**：`Sprint 1/`、`开发进度 - Development Progress.md`、`项目总结 - Project Summary.md`、`CookingLog/`、`Daily Log/` — 记录每个 Sprint 与每次大改动的背景与结论。  
  - **部署与运维**：`awsLog/1.12-自动化部署说明手册.md` — 前端如何改 Base URL、后端如何看 Actions、故障时如何查日志。  

- **作用**：减少口头传递、降低「只改一端导致联调失败」的概率，便于异步协作与交接。

### 4.3 实习深度洞察 (Internship Insights)

#### 从学生到准开发者的转变

- **安全**：  
  - 数据库密码、API Key 不落库、不提交仓库，通过环境变量与 GitHub Secrets 注入；JWT 配置独立、可更换。  

- **延迟与可用性**：  
  - 前端通过 `ApiConfig` 统一配置后端地址（真机/模拟器/AWS 公网 IP），避免硬编码；云端部署保证服务始终可访问，演示与联调不依赖本机。  

- **用户体验**：  
  - 前端在「真实生产环境」下考虑：加载状态、错误提示、离线/网络失败时的降级（如本地 YOLO + 云端 Gemini 双通道识别）；语音、手势、专注模式等围绕「烹饪场景」设计，体现从「能跑」到「好用」的思考。

## 第五部分：未来展望与建议 (7分钟)

### 5.1 项目反思：若重来一次，架构上可做的改进

- **前后端契约**：  
  - 更早引入 **OpenAPI/Swagger** 或从后端生成 API 文档/类型定义，减少手写文档与字段不一致。  

- **后端**：  
  - 将 DB 从 EC2 本机迁至 **RDS**，便于备份、扩缩容与高可用；对高 QPS 的接口考虑缓存（如 Redis）。  

- **前端**：  
  - 更早引入**状态管理**（如 Provider/Riverpod）和**命名路由**，便于复杂流程与深层链接。  

- **部署与质量**：  
  - 在 CI 中增加**自动化测试**（单元测试、API 测试），再部署；考虑多环境（dev/staging/prod）与分支策略，避免直接 Push 即覆盖生产。

---

### 5.2 未来建议

- **声控交互**：  
  - 前端已具备语音指令与 TTS（见 `frontend-app/项目服务总结.md`），可进一步与 AI 对话式推荐结合（如「根据今天冰箱里的菜推荐一道菜」）。  

- **社区分享**：  
  - 在现有「家庭」维度上，增加食谱/菜单的分享、发现与收藏，需要扩展 user/cooking 模型与权限设计。  

- **AI 推荐精准度**：  
  - 利用健康目标、历史摄入与偏好（已有 HealthGoal、FamilyMember 偏好、营养估算），做更个性化的热量与营养素控制；对生成结果做 A/B 或人工反馈收集，迭代 prompt 与模型选择。
